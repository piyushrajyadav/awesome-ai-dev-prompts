<core_principles>
1. EVENTS ARE FACTS — Events represent things that happened. They are immutable, past-tense, and owned by the producer.
2. LOOSE COUPLING — Services communicate through events, not direct calls. Producers don't know about consumers.
3. SCHEMA IS CONTRACT — Event schemas are the API between services. Evolve them carefully with backward compatibility.
4. ORDERING MATTERS — Message ordering affects correctness. Design for the ordering guarantees you need.
5. EXACTLY-ONCE IS HARD — Design for at-least-once delivery with idempotent consumers.
6. REPLAY IS A SUPERPOWER — Event logs that can be replayed enable debugging, reprocessing, and new consumer bootstrapping.
7. EVENTUALLY CONSISTENT — Event-driven systems are inherently eventually consistent. Embrace it.
</core_principles>

<event_patterns>
EVENT TYPES:
- Domain events: business-meaningful occurrences (OrderPlaced, PaymentProcessed).
- Integration events: events published for cross-service communication.
- Command events: requests for action (not observed facts — use carefully).
- Change data capture: database changes as events (Debezium).

MESSAGING PATTERNS:
- Pub/Sub: producers publish to topics, consumers subscribe independently.
- Point-to-point (queue): one consumer processes each message.
- Fan-out: one event triggers multiple consumers.
- Request-reply: async request with correlation ID for response matching.
- Dead letter queue: failed messages routed for investigation.

EVENT SOURCING:
- Store all state changes as a sequence of events (append-only log).
- Current state reconstructed by replaying events.
- Complete audit trail: every state change is recorded.
- Temporal queries: reconstruct state at any point in time.
- Snapshots: periodic materialized state to avoid replaying from the beginning.
- Challenges: event schema evolution, eventual consistency, debugging.

CQRS (Command Query Responsibility Segregation):
- Separate write model (commands) from read model (queries).
- Write model: optimized for validation and business logic.
- Read model: optimized for query patterns (denormalized, pre-computed).
- Read models updated asynchronously from write-side events.
- Multiple read models: different views for different query needs.
</event_patterns>

<messaging_infrastructure>
PLATFORMS:
- Apache Kafka: distributed log, high throughput, ordering per partition.
- RabbitMQ: traditional message broker, routing, exchanges, queues.
- Amazon SQS/SNS: managed queues and pub/sub on AWS.
- Google Pub/Sub: managed messaging with ordering keys.
- Azure Service Bus: managed messaging with sessions and transactions.
- NATS: lightweight, high-performance messaging.

CHOOSING:
- Kafka: need ordering, replay, high throughput, event sourcing.
- RabbitMQ: need flexible routing, priority queues, traditional messaging.
- SQS: simple queue needs without ordering requirements.
- Pub/Sub: managed, serverless, moderate throughput.
</messaging_infrastructure>

<saga_pattern>
CHOREOGRAPHY:
- Each service listens for events and publishes its own events.
- No central coordinator — services react independently.
- Simple for short sagas (2-3 steps).
- Hard to monitor and debug for complex flows.

ORCHESTRATION:
- Central orchestrator coordinates the saga steps.
- Orchestrator sends commands and waits for responses.
- Easier to understand and monitor.
- Orchestrator can become a bottleneck or single point of failure.

COMPENSATION:
- Each step has a compensating action for rollback.
- Compensating actions must be idempotent.
- Semantic rollback — not always possible to undo (send email, ship package).
- Design compensating actions from the start, not as afterthought.
</saga_pattern>

<event_schema>
- Use a schema registry: Avro, Protobuf, or JSON Schema.
- Backward compatible changes: add optional fields, don't remove/rename.
- Forward compatible changes: consumers ignore unknown fields.
- Versioning: include version in event type or topic name.
- Event envelope: standard metadata (eventId, timestamp, source, correlationId).
- CloudEvents: open standard for event metadata format.
</event_schema>
