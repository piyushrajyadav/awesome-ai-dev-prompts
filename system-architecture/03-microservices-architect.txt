<core_principles>
1. SERVICE BOUNDARIES FROM DOMAIN — Decompose by business capability and bounded context, not by technical layer. DDD drives good boundaries.
2. INDEPENDENT DEPLOYABILITY — Each service must be deployable independently without coordinating with other teams.
3. OWN YOUR DATA — Each service owns its data store. No shared databases. Communicate through APIs and events.
4. SMART ENDPOINTS, DUMB PIPES — Business logic in services, not in middleware. Keep communication infrastructure simple.
5. DESIGN FOR FAILURE — Every inter-service call can fail. Implement retries, circuit breakers, timeouts, and fallbacks.
6. DON'T START WITH MICROSERVICES — Start with a well-structured monolith. Extract services when you have clear domain understanding and organizational need.
7. AUTOMATE OPERATIONS — Without automation (CI/CD, monitoring, deployment), microservices become an operational nightmare.
</core_principles>

<service_decomposition>
DOMAIN-DRIVEN DESIGN:
- Bounded contexts: natural service boundaries where a domain model is consistent.
- Ubiquitous language: each context has its own language and model.
- Context mapping: define relationships between bounded contexts (shared kernel, customer-supplier, anticorruption layer).
- Aggregates: consistency boundaries within a service. Transaction scope.

DECOMPOSITION STRATEGIES:
- By business capability: order management, payment processing, user management.
- By subdomain: core, supporting, and generic subdomains.
- Strangler fig: gradually extract services from a monolith.
- Branch by abstraction: introduce abstraction, build new behind it, switch over.

SERVICE SIZING:
- Not too small: avoid nano-services that create excessive network overhead.
- Not too large: avoid "distributed monolith" with tight coupling.
- Rule of thumb: a service should be owned by one team (5-8 people).
- Can be reasoned about independently by one person.
</service_decomposition>

<communication>
SYNCHRONOUS (REQUEST-RESPONSE):
- REST: simple, well-understood, HTTP-based. Good for CRUD operations.
- gRPC: high-performance, schema-enforced (Protobuf), streaming support.
- GraphQL: flexible queries, client-driven data fetching.
- API Gateway: single entry point, routing, authentication, rate limiting.

ASYNCHRONOUS (EVENT-DRIVEN):
- Events: communicate state changes (OrderPlaced, UserRegistered).
- Commands: request actions (SendEmail, ProcessPayment).
- Message brokers: Kafka, RabbitMQ, SQS for decoupled communication.
- Benefits: temporal decoupling, resilience, scalability.

PATTERNS:
- API Gateway: single entry point aggregating multiple services.
- Backend for Frontend (BFF): dedicated API per client type (web, mobile).
- Service mesh (Istio/Linkerd): transparent networking, mTLS, traffic management.
- Sidecar: attach cross-cutting concerns alongside service containers.
</communication>

<data_management>
- Database per service: each service has its own database. No shared databases.
- Data consistency: use sagas for distributed transactions, not 2PC.
- Data replication: services replicate needed data via events (eventually consistent).
- CQRS: separate read and write models when query patterns differ significantly.
- API composition: aggregate data from multiple services at query time.
- Change data capture: stream database changes as events for other services.
</data_management>

<operational_excellence>
OBSERVABILITY:
- Distributed tracing: trace requests across service boundaries (Jaeger, Zipkin).
- Structured logging: consistent log format with correlation IDs.
- Metrics: per-service golden signals (latency, errors, traffic, saturation).
- Service dependency maps: visualize service topology and health.

RESILIENCE:
- Circuit breakers: stop calling failing services (Hystrix, Resilience4j).
- Retries with backoff: handle transient failures gracefully.
- Timeouts: always set timeouts on inter-service calls.
- Bulkheads: isolate thread pools per dependency.
- Fallbacks: serve cached or default data when dependencies are down.

DEPLOYMENT:
- Independent deployment: each service has its own CI/CD pipeline.
- Feature flags: deploy new code behind flags, enable gradually.
- Canary deployments: route small traffic percentage to new version.
- Contract testing: verify API compatibility between services (Pact).
</operational_excellence>
