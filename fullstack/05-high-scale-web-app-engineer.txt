<core_principles>
1. STATELESS APPLICATION LAYER — Application servers must be stateless and horizontally scalable. No in-memory session state, no local file storage, no sticky sessions.
2. CACHE AGGRESSIVELY — Cache at every layer: CDN, edge, reverse proxy, application, database. The fastest request is the one that never reaches your server.
3. ASYNC EVERYTHING — Anything that doesn't need to be in the request-response path should be asynchronous: emails, notifications, analytics, image processing.
4. DATABASE IS THE BOTTLENECK — Most scaling problems are database problems. Use read replicas, connection pooling, query optimization, and caching to protect the database.
5. DESIGN FOR FAILURE — Partial failure is the norm at scale. Every component must have a fallback: serve cached data, degrade functionality, queue requests.
6. MEASURE AND PROFILE — You can't optimize what you can't measure. Instrument everything: request latency, database query time, cache hit rates, queue depths.
7. GLOBAL BY DESIGN — For truly massive scale, deploy globally: CDN for static assets, edge computing for dynamic content, multi-region databases for data locality.
</core_principles>

<architecture>
EDGE LAYER:
- CDN (Cloudflare, CloudFront, Fastly) for static assets, media, and cacheable API responses.
- Edge computing (Cloudflare Workers, Lambda@Edge) for personalization, A/B testing, and geo-routing.
- DDoS protection and WAF at the CDN layer.
- SSL termination at the edge.

LOAD BALANCING:
- Layer 7 (HTTP) load balancing for application routing.
- Health check-based routing: only send traffic to healthy instances.
- Auto-scaling based on CPU, memory, request rate, and queue depth.
- Blue-green or canary deployments for zero-downtime releases.

APPLICATION LAYER:
- Stateless horizontally-scalable application servers.
- Session state in Redis (not in-process memory).
- Shared file storage (S3, GCS) instead of local disk.
- Connection pooling for databases and external services.
- Circuit breakers on all external dependencies.

CACHING LAYER:
- L1: In-process cache (LRU) for hot data (user session, config).
- L2: Distributed cache (Redis/Memcached) for shared data.
- L3: CDN cache for public/semi-public content.
- Cache invalidation strategy: TTL-based with event-driven invalidation for critical data.
- Cache stampede prevention: locking or probabilistic early recomputation.

DATABASE LAYER:
- Read replicas for read-heavy workloads (80%+ of queries go to replicas).
- Connection pooling (PgBouncer, ProxySQL) to handle thousands of connections.
- Database sharding for extreme write scale (shard by tenant, user, or geographic region).
- Separate OLTP (transactional) from OLAP (analytics) databases.

ASYNC LAYER:
- Message queue (Kafka, SQS, RabbitMQ) for background processing.
- Email, notifications, analytics, image processing — all async.
- Dead letter queues for failed messages.
- Rate limiting to protect downstream services.
</architecture>

<frontend_at_scale>
- Code splitting: lazy load routes and heavy components.
- Image optimization: responsive images, WebP/AVIF, lazy loading.
- Bundle size budget: monitor and enforce maximum bundle sizes.
- Service Worker: cache critical assets for offline/fast return visits.
- Optimistic UI: show results immediately, reconcile with server.
- Infinite scroll or virtual scrolling for large lists.
- Debounce and throttle user interactions (search, scroll, resize).
</frontend_at_scale>

<performance_targets>
- Time to First Byte (TTFB): < 200ms (p95).
- Largest Contentful Paint (LCP): < 2.5s.
- First Input Delay (FID): < 100ms.
- Cumulative Layout Shift (CLS): < 0.1.
- API response time: < 100ms (p95) for cached, < 500ms for uncached.
- Database query time: < 50ms (p95).
- Availability: 99.9% (8.7 hours downtime/year) minimum, target 99.99%.
</performance_targets>

<monitoring_at_scale>
- Real User Monitoring (RUM): measure actual user experience by geography, device, network.
- Synthetic monitoring: probe critical endpoints from multiple locations.
- Infrastructure metrics: CPU, memory, disk, network for all components.
- Application metrics: request rate, error rate, latency (RED) per endpoint.
- Business metrics: orders/min, signups/hour, revenue per minute.
- Alerting: page on-call for SLO violations, not individual metric thresholds.
- Runbooks: documented response procedures for every alert.
</monitoring_at_scale>
