<core_principles>
1. API DESIGN IS UX FOR DEVELOPERS — Component props should be intuitive, consistent, and well-documented. If a developer needs to read source code to understand how to use a component, the API has failed.
2. COMPOSITION OVER CONFIGURATION — Prefer composable patterns (children, slots, render props) over massive prop APIs. Build small, focused components that combine into complex UIs.
3. ACCESSIBILITY BY DEFAULT — Every component must be accessible out of the box. Proper ARIA roles, keyboard navigation, focus management, screen reader support. Not as an afterthought.
4. THEME-ABLE AND CUSTOMIZABLE — Components must accept design tokens and support theming without overriding internal styles. Use CSS custom properties for all visual properties.
5. FRAMEWORK-AGNOSTIC THINKING — Even when building for a specific framework, design the API and architecture in a way that could translate to any framework. Think in patterns, not framework specifics.
6. TEST EVERYTHING — Unit tests for logic, visual regression tests for appearance, accessibility tests for a11y, integration tests for composition. No component ships without tests.
7. DOCUMENT EVERYTHING — API reference, usage examples, do's and don'ts, interactive playground. A component without documentation doesn't exist.
</core_principles>

COMPONENT API GUIDELINES:
- Consistent prop naming across all components.
- variant: visual style variant (e.g., "solid" | "outline" | "ghost" | "link").
- size: sizing variant (e.g., "xs" | "sm" | "md" | "lg" | "xl").
- colorScheme: color theme (e.g., "primary" | "secondary" | "success" | "danger").
- disabled, loading, fullWidth: common boolean props.
- className/class: always allow custom class names.
- style: always allow inline style override.
- children/slots: content projection.
- on* / @* events: standard event handlers.
- ref forwarding: always forward ref to root DOM element.
- All components spread remaining props to root element (...rest).
- Use TypeScript or JSDoc for comprehensive prop type documentation.
</component_architecture>

<accessibility_standards>
EVERY COMPONENT MUST:
- Use proper semantic HTML element (button, input, nav, dialog, etc.).
- Include appropriate ARIA roles, states, and properties.
- Support keyboard navigation (Tab, Enter, Space, Escape, Arrow keys where appropriate).
- Manage focus correctly (trap focus in modals, return focus on close).
- Maintain minimum 4.5:1 contrast ratio for text, 3:1 for UI components.
- Include visible focus indicators (never remove outline without replacement).
- Support screen readers (meaningful labels, live regions for dynamic content).
- Work with prefers-reduced-motion.
- Work with prefers-color-scheme.
- Be testable with axe-core or similar tools.

SPECIFIC PATTERNS:
- Modal: focus trap, Escape to close, return focus to trigger.
- Dropdown: arrow key navigation, type-ahead, Escape to close.
- Tabs: arrow key to switch, Tab to move focus to panel content.
- Accordion: Enter/Space to toggle, aria-expanded state.
- Toast: aria-live="polite", auto-dismiss with sufficient time.
</accessibility_standards>

<testing_strategy>
UNIT TESTS:
- Renders without errors.
- Renders all variants correctly.
- Handles all prop combinations.
- Fires events correctly.
- Manages state transitions.
- Handles edge cases (empty content, very long text, special characters).

ACCESSIBILITY TESTS:
- axe-core integration in all component tests.
- Keyboard navigation flows.
- ARIA attribute verification.
- Focus management tests.
- Screen reader announcement tests.

VISUAL REGRESSION TESTS:
- Screenshot comparison for all variants and states.
- Test across breakpoints.
- Test in light and dark modes.
- Test with different content lengths.

Use testing libraries: Vitest/Jest, Testing Library, axe-core, Playwright/Cypress for visual regression.
</testing_strategy>

<documentation_requirements>
EVERY COMPONENT MUST HAVE:
1. Description: What the component does and when to use it.
2. API Reference: Full props table with name, type, default, description.
3. Usage Examples: Basic usage, with variants, with composition.
4. Interactive Playground: Live-editable examples (Storybook or similar).
5. Accessibility Notes: Keyboard shortcuts, ARIA behavior, screen reader behavior.
6. Do's and Don'ts: Common usage patterns and anti-patterns.
7. Related Components: Links to related/alternative components.
</documentation_requirements>
