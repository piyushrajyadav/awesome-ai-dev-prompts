<core_principles>
1. SERVER COMPONENTS BY DEFAULT — Assume every component is a Server Component unless it absolutely needs client interactivity. RSCs reduce bundle size and keep secrets on the server.
2. PUSH 'USE CLIENT' DOWN THE TREE — The client boundary (`"use client"`) should be as far down the component tree as possible. Only make the interactive leaves client components; keep the layout and data-fetching high up on the server.
3. DATA FETCHING ON THE SERVER — Fetch data directly in Server Components using native `fetch` or ORMs. Do not use `useEffect` or client-side fetching libraries unless real-time or highly dynamic client data is needed.
4. CACHING IS EXPLICIT — Understand that Next.js aggressively caches everything by default. You must explicitly opt-out of caching for dynamic routes or mutate the cache using `revalidatePath` or `revalidateTag`.
5. ACTIONS OVER APIS — Use Server Actions for mutations instead of building REST API routes. Co-locate data mutation logic with the forms or components that trigger them.
6. STREAMING AND SUSPENSE — Wrap slow data-fetching components in `<Suspense>` boundaries to stream the UI to the client instantly while data resolves.
7. EDGE FIRST — Write code that can run on the Edge runtime whenever possible for optimal global performance, being mindful of Node.js API limitations at the Edge.
</core_principles>

<app_router_architecture>
FILE-SYSTEM ROUTING:
- `page.tsx`: The UI unique to a route (Server Component).
- `layout.tsx`: Shared UI that wraps pages or nested layouts. State is preserved on navigation.
- `loading.tsx`: An instant fallback UI (Suspense boundary) shown while a route loads.
- `error.tsx`: Error boundary UI (Must be a Client Component).
- `not-found.tsx`: Fallback UI for 404s.
- `route.ts`: API endpoints (Route Handlers).
- `default.tsx`: Fallback UI for Parallel Routes.

ROUTE SEGMENTS:
- `app/dashboard/page.tsx` → `/dashboard`
- `app/blog/[slug]/page.tsx` → `/blog/hello-world` (Dynamic)
- `app/shop/[...slug]/page.tsx` → `/shop/clothes/tops` (Catch-all)
- `app/(marketing)/about/page.tsx` → `/about` (Route Groups for logical organization)
- `app/@modal/(.)photo/[id]/page.tsx` → Intercepting and Parallel Routes for complex patterns like modals over feeds.
</app_router_architecture>

<data_fetching_and_caching>
SERVER COMPONENT FETCHING:
```tsx
// This component runs on the server. No hooks needed.
export default async function UserProfile({ params }: { params: { id: string } }) {
  // fetch is extended by Next.js to memoize requests
  const res = await fetch(`https://api.example.com/users/${params.id}`);
  const user = await res.json();
  return <div>{user.name}</div>;
}
```

CACHE CONTROL:
- Default: `fetch('...', { cache: 'force-cache' })` — Static, cached indefinitely.
- Dynamic: `fetch('...', { cache: 'no-store' })` — Fetched on every request.
- Revalidation (ISR): `fetch('...', { next: { revalidate: 3600 } })` — Cached, revalidated every hour.
- On-Demand Revalidation: `fetch('...', { next: { tags: ['user'] } })` — Revalidated via `revalidateTag('user')`.

DATA PASSING RULES:
- Server Components can pass data to Client Components as props.
- Props passed from Server to Client must be serializable (no functions, dates must be strings, etc.).
- Client Components CANNOT import Server Components. Instead, they must accept Server Components as `children` or explicit props.
</data_fetching_and_caching>

<server_actions>
MUTATIONS (Next.js 14+):
- Use Server Actions for handling form submissions and data mutations without building API routes.
- Define them with the `"use server"` directive.

```tsx
// app/actions.ts
"use server"
import { revalidatePath } from 'next/cache';

export async function updateUser(formData: FormData) {
  const name = formData.get('name');
  // Mutate database...
  revalidatePath('/profile'); // Purge cache 
}

// In a Client or Server Component:
import { updateUser } from './actions';
// <form action={updateUser}>
```

- Combine with `useFormStatus` (to show pending states in UI) and `useFormState` (to handle success/error messages).
</server_actions>

<optimization_strategies>
IMAGES:
- Always use `next/image` for automatic optimization, WebP/AVIF sizing, and lazy loading.
- Set `width` and `height` (or `fill` with `sizes`) to prevent CLS.
- Provide `blurDataURL` or use `placeholder="blur"` for static imports.

FONTS:
- Use `next/font/google` or `next/font/local` to automatically self-host fonts and prevent layout shift.

METADATA (SEO):
- Use the Metadata API in Server Components (`export const metadata: Metadata = {...}`).
- Use `generateMetadata({ params })` for dynamic SEO (e.g., blog post titles).
- Generate dynamic sitemaps and opengraph images using `sitemap.ts` and `opengraph-image.tsx`.

STATIC EXPORT:
- Understand when to use `output: 'export'` for full static site generation (SSG), knowing its limitations (no Server Actions, no dynamic routes without `generateStaticParams`).
</optimization_strategies>

<common_pitfalls_to_avoid>
- DO NOT use `"use client"` at the top of the app tree (e.g., `layout.tsx`). This defeats the purpose of the App Router.
- DO NOT use Context providers for state that can be passed as props or read directly from the URL.
- DO NOT fetch data in `useEffect` if it can be fetched in a Server Component.
- BEWARE of "Route Cache" staleness. If navigation doesn't show updated data, you forgot to use `revalidatePath` or `router.refresh()`.
- DO NOT leak secrets. Use `server-only` package to mark files that contain sensitive DB logic to prevent accidental client imports. Environment variables must be prefixed with `NEXT_PUBLIC_` to be readable on the client.
</common_pitfalls_to_avoid>
