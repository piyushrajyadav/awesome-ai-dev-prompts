<vue3_architecture>
SCRIPT SETUP PARADIGM:
- Always use `<script setup lang="ts">`.
- Avoid the traditional `setup()` function unless explicitly required by a specific architectural edge case.
- Variables and functions declared at the top level of `<script setup>` are automatically exposed to the template. No need to `return` them.

PROPS AND EMITS:
- Use TS-based declaration for props: `const props = defineProps<{ id: string, items: Array<Item> }>()`.
- Use `withDefaults` for default values: `withDefaults(defineProps<Props>(), { items: () => [] })`.
- Use TS for emits: `const emit = defineEmits<{ (e: 'update', id: number): void, (e: 'delete'): void }>()`.

V-MODEL AND TWO-WAY BINDING:
- In Vue 3, `v-model` uses `modelValue` as the prop and `@update:modelValue` as the event.
- Use multiple v-models naming them: `v-model:title="title"` translates to `title` prop and `@update:title` event.
- Prefer Vue 3.4+ `defineModel()` macro for cleaner v-model implementation.
```vue
<script setup>
const count = defineModel<number>('count', { default: 0 })
// mutating count.value automatically emits the update event
</script>
```
</vue3_architecture>

<reactivity_mastery>
REF VS REACTIVE:
- `ref`: Use for primitives (string, boolean, number) OR when you need to reassign an entire object/array.
- `reactive`: Use for grouping related state into a single deep proxy object. Cannot be reassigned without losing reactivity. Can't be destructured without `toRefs`.
- General rule of thumb: Default to `ref` for consistency, unless modeling a specific complex domain object.

COMPUTED AND WATCHERS:
- `computed`: Use for derived state. Must be pure functions. Computed properties are cached based on their reactive dependencies.
- `watch`: Use for side effects in response to specific state changes (e.g., fetching data when an ID changes). Provide exact sources.
- `watchEffect`: Use when you want the effect to run immediately and automatically track any reactive data used inside it. Good for generic synchronization.

LIFECYCLE HOOKS:
- `onMounted`: DOM is ready. Good for canvas APIs, 3rd party chart libraries.
- `onUnmounted`: Crucial for cleanup (clearing intervals, removing custom event listeners) to prevent memory leaks.
</reactivity_mastery>

<composables_pattern>
- A composable is a function whose name starts with "use" extending the Composition API.
- It can use Vue's reactivity system internally.
- Example structure:
```ts
export function useMouse() {
  const x = ref(0)
  const y = ref(0)
  
  function update(event: MouseEvent) {
    x.value = event.pageX
    y.value = event.pageY
  }
  
  onMounted(() => window.addEventListener('mousemove', update))
  onUnmounted(() => window.removeEventListener('mousemove', update))
  
  // Expose state as refs
  return { x, y }
}
```
- Important: Always return `ref`s from composables, not reactive objects, so consumers can destructure them safely without losing reactivity.
</composables_pattern>

<pinia_state_management>
- Define stores using the Setup Store syntax (it perfectly mirrors the Composition API):
```ts
export const useCounterStore = defineStore('counter', () => {
  const count = ref(0) // State
  const doubleCount = computed(() => count.value * 2) // Getter
  function increment() { count.value++ } // Action
  
  return { count, doubleCount, increment }
})
```
- Pinia supports HMR, SSR, and DevTools out of the box.
- Do not use Pinia for server state (server syncing). Use tools like Vue Query (TanStack) or Nuxt's `useFetch` for API data management. Pinia is for client UI state.
</pinia_state_management>

<template_and_dom>
- Use `<template v-for>` to avoid unnecessary wrapper divs. Provide a `key` bound to a unique ID, never the array index if the list can be modified.
- Use `v-show` for elements toggled frequently (cheap toggle, high memory cost kept in DOM).
- Use `v-if` for elements that might never render or are structurally complex (expensive toggle, low memory cost since not in DOM).
- Use Vue's Transition element `<Transition name="fade">` for smooth entry/leave animations.
- Use `<Teleport to="body">` for Modals/Dialogs to escape z-index and overflow clipping issues of parent elements.
</template_and_dom>
