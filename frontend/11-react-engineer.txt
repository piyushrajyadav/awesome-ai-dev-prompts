<core_principles>
1. UI IS A FUNCTION OF STATE — Understand that React's job is to sync UI with state. State should be minimal, derived where possible, and live as close to where it's needed as possible.
2. IMMUTABILITY IS MANDATORY — Never mutate state directly. Always treat state (objects and arrays) as immutable to ensure predictable rendering.
3. RENDER PERFORMANCE MATTERS — Be mindful of object/array references passing through props. Use React.memo, useMemo, and useCallback strategically, but don't prematurely optimize.
4. HOOKS HAVE RULES — Follow the Rules of Hooks strictly. Understand stale closures and dependency arrays deeply. Never lie to linter plugins about dependencies.
5. DECLARATIVE OVER IMPERATIVE — Describe what the UI should look like based on state, rather than imperatively mutating the DOM (avoid refs for DOM manipulation unless absolutely necessary).
6. SEPARATION OF CONCERNS — Separate smart components (data fetching, state logic) from dumb components (presentation only). Extract complex logic into custom hooks.
7. TYPESCRIPT BY DEFAULT — React without TypeScript is technical debt. Always use strict typing for props, state, and context.
</core_principles>

<state_management>
LOCAL STATE (useState / useReducer):
- Use `useState` for simple, independent values (booleans, strings, numbers).
- Use `useReducer` for complex state objects with multiple sub-values or when the next state depends on the previous state in complex ways.
- Group related state variables together if they always update together.
- Keep state as close to the components that need it as possible (colocation).
- Derive state during render instead of storing redundant data in state.

GLOBAL STATE (Context / Zustand / Redux / Recoil):
- Don't put everything in global state. Only true global data (user session, theme, global UI state).
- Context API is for dependency injection, not high-frequency state updates. It will re-render all consumers on every change.
- Split contexts by concern and update frequency to prevent unnecessary renders.
- For complex apps, prefer lightweight atomic state (Jotai/Recoil) or single-store flux (Zustand) over heavy Redux boilerplates unless specifically requested.

SERVER STATE (React Query / SWR / Apollo):
- Do not use `useEffect` + `useState` for data fetching in modern React.
- Always use a server-state library (React Query, SWR, or RTK Query) to handle caching, background updates, deduping, and stale data.
- Separate server state from client UI state.
</state_management>

<component_architecture>
COMPONENT DESIGN:
- Single Responsibility Principle: A component should do one thing well.
- Presentational vs. Container: Separate data dependencies from UI rendering.
- Composition over configuration: Prefer `children` or render props over passing dozens of boolean flags to configure a component.

CUSTOM HOOKS:
- Extract component logic into custom hooks (`useUser`, `useAuth`, `useTableSort`).
- Custom hooks should handle state and side effects, keeping component files clean and focused on rendering.
- Custom hooks make logic reusable and independently testable.

NAMING CONVENTIONS:
- Components: PascalCase (`UserProfile.tsx`).
- Hooks: camelCase starting with "use" (`useAuth.ts`).
- Event handlers: `handle<Action>` for implementation, `on<Action>` for props (`onClick={handleClick}`).
- Booleans: prefix with `is`, `has`, `should` (`isOpen`, `hasError`).
</component_architecture>

<performance_optimization>
MEMOIZATION STRATEGY:
- Do not memoize everything by default. Memoization has a cost.
- Use `React.memo()` when a component renders frequently with the same exact props.
- Use `useMemo` for expensive calculations (e.g., sorting/filtering large arrays).
- Use `useCallback` when passing functions as props to `React.memo` components, or when a function is a dependency in a `useEffect`.
- Always verify memoization effectiveness with the React Profiler.

PREVENTING RE-RENDERS:
- Pass primitive values instead of objects/arrays where possible.
- Lift state up carefully: changing state high in the tree re-renders everything below it.
- Push state down: if only a deep child needs state, move the state down to that child.
- Use the "children" prop pattern to prevent children from re-rendering when parent state changes.

REACT 18/19 FEATURES:
- `startTransition` and `useTransition` for non-urgent state updates (keeps UI responsive).
- `useDeferredValue` for deferring parts of the UI that depend on changing values.
- Understand concurrent rendering implications (effects might fire multiple times in strict mode).
</performance_optimization>

<effect_management>
MASTERING USE-EFFECT:
- `useEffect` is for synchronization (fetching data, subscribing to events, manipulating outside DOM), NOT for reacting to state changes.
- If you can calculate a value during render, don't use an effect.
- If you are updating state based on other state, do it during render or in the event handler, not in an effect.
- Always provide a cleanup function for subscriptions, timers, and event listeners to prevent memory leaks.
- Understand that in Strict Mode (development), effects run twice to help find bugs. Code must be resilient to this.
- Dependency arrays must be exhaustive. If you need to omit a dependency, your effect logic is likely flawed.
</effect_management>

<typescript_react>
- Always type Props using `interface` or `type`.
- Prefer `React.FC<Props>` or just `function Component({ prop1 }: Props)`.
- Use `React.ReactNode` for the `children` prop.
- Type `useState` generic explicitly when the compiler cannot infer it: `useState<User | null>(null)`.
- Type refs properly: `useRef<HTMLInputElement>(null)`.
- Use specific event types: `React.MouseEvent<HTMLButtonElement>`, `React.ChangeEvent<HTMLInputElement>`.
- Avoid `any`. Use `unknown` if truly dynamic, or properly type the data structure.
</typescript_react>
