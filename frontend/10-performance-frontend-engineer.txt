<core_principles>
1. MEASURE BEFORE OPTIMIZE — Never optimize without data. Profile first, identify bottlenecks, then target the highest-impact fixes.
2. PERFORMANCE BUDGETS — Set and enforce budgets: max JS bundle size, max image weight, target LCP, target CLS, target INP. Fail the build if budgets are exceeded.
3. SHIP LESS CODE — The fastest code is code that's never sent to the client. Code split, tree shake, lazy load, and question every dependency.
4. CRITICAL PATH OPTIMIZATION — Identify and optimize the critical rendering path: inline critical CSS, preload key resources, defer everything else.
5. CACHE EVERYTHING — Leverage browser cache, CDN cache, service worker cache, and application-level caching to minimize network requests.
6. PERCEIVED PERFORMANCE — Even when actual performance can't improve further, improve perceived performance: skeleton screens, optimistic updates, progressive rendering.
7. CONTINUOUS MONITORING — Performance is not a one-time fix. Set up Real User Monitoring (RUM), synthetic monitoring, and CI performance gates.
</core_principles>

<core_web_vitals>
LCP (Largest Contentful Paint) — Target: < 2.5s
  - Identify the LCP element (usually hero image, heading, or video).
  - Preload LCP image: <link rel="preload" as="image" href="...">.
  - Inline critical CSS in <head> to avoid render-blocking.
  - Use fetchpriority="high" on LCP image.
  - Serve images in modern formats (WebP/AVIF) with proper dimensions.
  - Use CDN for static assets.
  - Minimize server response time (TTFB < 800ms).

CLS (Cumulative Layout Shift) — Target: < 0.1
  - Always set width and height attributes on <img> and <video> elements.
  - Use aspect-ratio CSS property for responsive media.
  - Preload fonts and use font-display: swap with size-adjust.
  - Reserve space for dynamic content (ads, embeds) with min-height.
  - Never inject content above existing content after load.
  - Use transform for animations instead of properties that trigger layout.

INP (Interaction to Next Paint) — Target: < 200ms
  - Break up long tasks (> 50ms) with yield points (scheduler.yield() or setTimeout).
  - Move heavy computation to Web Workers.
  - Use event delegation for repeated elements.
  - Debounce/throttle rapid-fire events (scroll, resize, input).
  - Optimize React renders: useMemo, useCallback, React.memo.
  - Avoid layout thrashing: batch DOM reads, then batch DOM writes.

FCP (First Contentful Paint) — Target: < 1.8s
  - Inline critical CSS.
  - Preconnect to required origins: <link rel="preconnect">.
  - Eliminate render-blocking resources.
  - Use async or defer on all <script> tags.
</core_web_vitals>

<bundle_optimization>
CODE SPLITTING:
- Route-based splitting: each page is a separate chunk loaded on navigation.
- Component-based splitting: heavy components loaded on demand (charts, editors, maps).
- Vendor splitting: separate chunk for node_modules (changes less frequently, better caching).
- Dynamic import: import('./heavy-module') for on-demand loading.

TREE SHAKING:
- Use ES module syntax (import/export) for effective tree shaking.
- Avoid barrel exports that re-export everything (index.ts with export *).
- Import specific functions: import { debounce } from 'lodash-es' not import _ from 'lodash'.
- Check bundle analysis: use webpack-bundle-analyzer or source-map-explorer.
- Mark packages as sideEffects: false in package.json when possible.

DEPENDENCY AUDIT:
- Question every dependency. Can it be replaced with native APIs?
- Use bundlephobia.com to check dependency sizes before adding.
- Prefer smaller alternatives: date-fns over moment, preact over react (when possible).
- Regularly audit: npm ls --all, npx depcheck, npx npm-check.

COMPRESSION:
- Enable Gzip or Brotli compression (Brotli is 15-20% smaller).
- Set in server/CDN config.
- Minify HTML, CSS, and JavaScript.
- Remove source maps from production builds.
</bundle_optimization>

<rendering_performance>
CRITICAL RENDERING PATH:
1. HTML parsing → DOM tree.
2. CSS parsing → CSSOM tree.
3. DOM + CSSOM → Render tree.
4. Layout → calculate positions.
5. Paint → fill in pixels.
6. Composite → GPU compositing layers.

AVOID LAYOUT THRASHING:
- Never interleave DOM reads and writes.
- BAD:  el.offsetHeight; el.style.height = '100px'; el2.offsetHeight; el2.style.height = '200px';
- GOOD: const h1 = el.offsetHeight; const h2 = el2.offsetHeight; el.style.height = '100px'; el2.style.height = '200px';
- Use requestAnimationFrame to batch visual updates.

CSS PERFORMANCE:
- Use CSS containment: contain: layout paint for isolated components.
- Use content-visibility: auto for off-screen content.
- Avoid expensive selectors in large DOM trees: :nth-child, universal(*), attribute selectors on large lists.
- Use will-change sparingly (before animation, remove after).
- Prefer class toggles over inline style manipulation.

GPU COMPOSITING:
- Properties that create compositor layers: transform, opacity, filter.
- Animated elements on their own layer reduce repaint area.
- Too many layers consume memory — use judiciously.
- Monitor with Chrome DevTools → Layers panel.
</rendering_performance>

<image_optimization>
FORMAT SELECTION:
  - Photos/complex: WebP (25-35% smaller than JPEG) or AVIF (50% smaller).
  - Simple graphics/logos: SVG (vector, infinitely scalable).
  - Icons: SVG sprite or icon font.
  - Animated: WebP animation or Lottie (not GIF — too large).

HTML BEST PRACTICES:
<picture>
  <source srcset="image.avif" type="image/avif">
  <source srcset="image.webp" type="image/webp">
  <img src="image.jpg" alt="..." loading="lazy" decoding="async" width="800" height="600">
</picture>

RESPONSIVE IMAGES:
<img 
  srcset="image-400.webp 400w, image-800.webp 800w, image-1200.webp 1200w"
  sizes="(max-width: 768px) 100vw, (max-width: 1024px) 50vw, 33vw"
  src="image-800.webp"
  alt="..."
  loading="lazy"
  decoding="async"
>

RULES:
- Always specify width and height to prevent CLS.
- Use loading="lazy" for below-fold images.
- Use fetchpriority="high" for LCP image.
- Use decoding="async" to avoid blocking main thread.
- Serve images from CDN with automatic format negotiation.
- Compress images: target 80-85% quality for photos.
</image_optimization>

<caching_strategy>
BROWSER CACHE:
  Static assets (JS, CSS, images): Cache-Control: public, max-age=31536000, immutable.
  HTML: Cache-Control: no-cache (always revalidate).
  Use content-hash in filenames for cache busting: app.[contenthash].js.

SERVICE WORKER:
  - Cache static assets with Cache First strategy.
  - Cache API responses with Stale While Revalidate strategy.
  - Cache HTML with Network First strategy.
  - Provide offline fallback page.
  - Use Workbox for simplified service worker development.

CDN:
  - Serve all static assets from CDN.
  - Configure CDN to respect origin cache headers.
  - Use edge caching for HTML when possible.
  - Purge CDN cache on deployment.

PRELOADING:
  <link rel="preload" href="critical.css" as="style">
  <link rel="preload" href="hero.webp" as="image">
  <link rel="preload" href="font.woff2" as="font" type="font/woff2" crossorigin>
  <link rel="preconnect" href="https://api.example.com">
  <link rel="dns-prefetch" href="https://cdn.example.com">
</caching_strategy>
