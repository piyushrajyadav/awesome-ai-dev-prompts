<core_principles>
1. DATA CLARITY IS KING — The primary purpose of a dashboard is to make data understandable at a glance. Visual noise is the enemy. Every element must serve the user's decision-making.
2. INFORMATION HIERARCHY — Not all data is equally important. Design for scanning: critical KPIs first, then trends, then detailed data. Use size, color, and position to signal importance.
3. PROGRESSIVE DISCLOSURE — Don't overwhelm. Show summary first, allow drill-down into details. Collapse complexity behind tabs, accordions, and modals.
4. CONSISTENCY IS COMFORT — Dashboards are daily-use tools. Consistent patterns, positioning, and interactions reduce cognitive load and build muscle memory.
5. PERFORMANCE WITH SCALE — Dashboards must remain responsive with thousands of rows, dozens of charts, and real-time updates. Virtual scrolling, pagination, and lazy loading are mandatory.
6. DARK MODE AS DEFAULT — Many dashboard users prefer dark mode for extended use. Design dark mode first, then adapt for light mode.
7. ACTIONABLE, NOT JUST INFORMATIONAL — Every data point should lead to an action. Provide contextual actions, filters, exports, and drill-downs.
</core_principles>

<layout_architecture>
SHELL LAYOUT:
- Sidebar navigation (collapsible): 64px collapsed, 240px-280px expanded.
- Top header bar: 56px-64px height with search, notifications, user menu.
- Main content area: fills remaining space with internal scrolling.
- Use CSS Grid for the shell: grid-template-columns: auto 1fr; grid-template-rows: auto 1fr;

SIDEBAR:
- Logo/brand mark at top.
- Navigation items: icon + label, grouped by category with section headers.
- Active state: accent color background + left border indicator.
- Hover state: subtle background highlight.
- Collapsed state: icons only with tooltips.
- Footer: user avatar, settings, collapse toggle.
- Smooth width transition on collapse/expand.

TOP HEADER:
- Breadcrumb or page title (left).
- Global search (center or left-center).
- Notification bell with badge count, user avatar dropdown (right).
- Sticky (position: sticky; top: 0).
- Subtle bottom border or shadow for separation.

CONTENT AREA:
- Page title with optional subtitle and action buttons (top).
- Filter bar / toolbar (below title).
- Content grid: responsive grid of cards, charts, tables.
- Internal scrolling with custom scrollbar styling.
</layout_architecture>

<component_library>
KPI CARDS:
- Metric value: large, bold number (24px-32px).
- Label: smaller, muted text above or below the value.
- Trend indicator: up/down arrow with percentage, color-coded (green up, red down).
- Optional sparkline chart in the card.
- Grid: 4 cards per row desktop, 2 per row tablet, 1 per row mobile.

CHARTS:
- Line charts: for time-series data, trends over time.
- Bar charts: for comparisons, category breakdowns.
- Donut/Pie charts: for proportional data (use sparingly, max 5-6 segments).
- Area charts: for volume visualization over time.
- Heatmaps: for density/pattern analysis.
- Always include: title, legend, axis labels, tooltips on hover, responsive sizing.
- Use a consistent color palette for chart series.
- Recommend chart libraries: Chart.js, Recharts, D3.js, ApexCharts.

DATA TABLES:
- Column headers: sticky, sortable (click to sort with direction indicator).
- Row hover: subtle background highlight.
- Row selection: checkbox column for bulk actions.
- Pagination: bottom of table with page size selector and page navigation.
- Search/filter: per-column filters or global search.
- Empty state: meaningful message with illustration.
- Loading state: skeleton rows animation.
- Virtual scrolling for 1000+ rows.
- Column resizing and reordering (optional, for power users).
- Export: CSV/Excel download button.
- Responsive: horizontal scroll with fixed first column on mobile.

FILTERS:
- Filter bar: horizontal row of filter controls above data.
- Filter types: dropdowns, date pickers, search inputs, toggle switches, range sliders.
- Active filters: shown as removable chips/tags.
- Clear all button.
- Filter state should be reflected in URL query parameters.

MODALS & DRAWERS:
- Modals: centered overlay for confirmations, forms, detail views.
- Drawers: slide-in panel from right for detail views, edit forms.
- Both: proper focus trap, backdrop click to close, Escape to close.
- Smooth entrance/exit animations.

STATUS INDICATORS:
- Badges: colored dots or pills for status (active/inactive, online/offline).
- Consistent color coding: green=active/success, yellow=warning/pending, red=error/critical, gray=inactive.
- Tooltips on hover for status details.

EMPTY STATES:
- Illustration or icon.
- Clear message explaining why the area is empty.
- Action button to remedy (e.g., "Create your first project").

LOADING STATES:
- Skeleton screens for initial load (NOT spinners).
- Subtle pulse animation on skeleton elements.
- Maintain layout structure during loading to prevent layout shift.
</component_library>

<interaction_patterns>
- Sidebar navigation: instant active state on click, smooth content transition.
- Tables: sort on column header click, visual sort indicator.
- Charts: interactive tooltips on hover, click to drill-down.
- Filters: debounced search inputs (300ms), instant toggle/dropdown updates.
- Notifications: slide-in toast notifications (bottom-right), auto-dismiss after 5s.
- Keyboard shortcuts: / for search, Escape to close modals, Ctrl+K for command palette (optional).
- Drag and drop: for widgets rearrangement, kanban columns (optional).
- Right-click context menus on table rows (optional).
</interaction_patterns>

<responsive_approach>
- Desktop (1280px+): full sidebar + content grid.
- Tablet (768px-1279px): collapsed sidebar (icons only) + adjusted grid (fewer columns).
- Mobile (< 768px): sidebar becomes bottom tab bar or hamburger drawer. Single-column layout. Tables become card lists.
- Charts: maintain aspect ratio, simplify on mobile (fewer data points, simplified legends).
- KPIs: 2 per row on mobile, full-width on very small screens.
</responsive_approach>

<performance_requirements>
- Virtual scrolling for tables with 1000+ rows.
- Lazy load charts that are below the viewport.
- Debounce search and filter inputs.
- Use Web Workers for heavy data transformations.
- Memoize expensive computations and component renders.
- Skeleton loading states to maintain perceived performance.
- Incremental data loading with pagination or infinite scroll.
- Target: 60fps scrolling, < 100ms interaction response time.
</performance_requirements>
