<core_principles>
1. CACHE THE RIGHT THINGS — Cache data that is expensive to compute, frequently accessed, and tolerant of staleness.
2. INVALIDATION IS THE HARD PART — Getting data into cache is easy. Knowing when to remove it is where bugs live.
3. STALENESS IS A FEATURE — Most data doesn't need to be real-time. Define acceptable staleness per data type.
4. MULTI-LAYER — Browser → CDN → API Gateway → Application → Database. Cache at every appropriate layer.
5. CONSISTENCY MODEL — Define and document the consistency guarantee for each cached data type.
6. THUNDERING HERD PROTECTION — When cache expires, don't let all requests hit the database simultaneously.
7. MONITOR CACHE HEALTH — Hit ratio, miss ratio, eviction rate, memory usage. Low hit ratio means your caching isn't working.
</core_principles>

<caching_layers>
CLIENT-SIDE CACHING:
- Browser cache: HTTP Cache-Control headers for static assets.
- Service Worker cache: offline-first, stale-while-revalidate patterns.
- Application cache: in-memory cache for API responses.
- Cache-Control directives: max-age, no-cache, no-store, private, public.
- ETags / If-None-Match: conditional requests for validation.

CDN & EDGE CACHING:
- Static assets: CSS, JS, images with long max-age and versioned URLs.
- API responses: cache public, non-personalized API responses.
- Edge compute: Cloudflare Workers, Lambda@Edge for dynamic caching logic.
- Purging: instant purge capabilities when content changes.
- Vary header: cache different versions based on request headers.

APPLICATION CACHING:
- In-process cache: fastest access (HashMap, LRU cache). Per-instance, not shared.
- Distributed cache: Redis, Memcached. Shared across instances.
- Cache-aside (lazy loading): read from cache, on miss read from DB and populate cache.
- Write-through: write to cache and DB simultaneously.
- Write-behind: write to cache, async write to DB. Fastest writes, risk of data loss.
- Read-through: cache fetches from DB on miss automatically.

DATABASE CACHING:
- Query cache: cache query results (often built into ORM).
- Materialized views: pre-computed query results stored in DB.
- Buffer pool: database's own memory cache for frequently accessed pages.
</caching_layers>

<distributed_caching>
REDIS:
- Data structures: strings, hashes, lists, sets, sorted sets, streams.
- Persistence: RDB snapshots, AOF log, or hybrid.
- Clustering: Redis Cluster for horizontal scaling and data partitioning.
- Sentinel: high availability with automatic failover.
- Pub/Sub: cache invalidation broadcasting.
- TTL: per-key expiration for automatic cache eviction.

MEMCACHED:
- Simple key-value cache. Pure in-memory (no persistence).
- Multi-threaded: better CPU utilization than single-threaded Redis.
- Consistent hashing: distribute keys across nodes.
- Best for: simple caching where persistence isn't needed.

SELECTION:
- Redis when: need data structures, persistence, Pub/Sub, scripting.
- Memcached when: simple key-value caching, multi-threaded performance.
</distributed_caching>

<cache_invalidation>
STRATEGIES:
- TTL (Time-To-Live): simplest. Set expiration on cache entries.
- Event-based: invalidate on write events (database triggers, application events).
- Write-through invalidation: update cache on every write.
- Version-based: store version numbers, invalidate when version changes.
- Tag-based: tag cache entries, purge all entries with a tag.

PATTERNS:
- Stale-while-revalidate: serve stale data immediately, refresh in background.
- Cache stampede protection: lock on miss so only one request fetches from DB.
- Probabilistic early expiration: randomly refresh before TTL to spread load.
- Background refresh: preemptively refresh popular cache entries before expiry.

THUNDERING HERD SOLUTIONS:
- Lock/mutex: single request fetches, others wait.
- Request coalescing: deduplicate concurrent requests for the same key.
- Stale-while-revalidate: serve stale until refreshed.
- Probabilistic early recomputation: each request has a chance of refreshing early.
</cache_invalidation>

<eviction_policies>
- LRU (Least Recently Used): evict the least recently accessed entry. Most common.
- LFU (Least Frequently Used): evict the least frequently accessed entry.
- TTL: evict entries that have expired.
- Random: randomly evict entries. Simple, surprisingly effective.
- FIFO: evict the oldest entry. Simple but not access-aware.
- Redis: allkeys-lru, volatile-lru, allkeys-lfu, volatile-ttl, noeviction.
</eviction_policies>
