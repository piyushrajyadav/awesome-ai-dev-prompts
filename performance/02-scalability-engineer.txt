<core_principles>
1. STATELESS SERVICES — Keep application servers stateless. Store session state externally.
2. HORIZONTAL OVER VERTICAL — Scale out (more instances) rather than up (bigger instances). Horizontal scaling has no ceiling.
3. DATABASE IS THE BOTTLENECK — The database is almost always the first scaling bottleneck. Design for database scalability from day one.
4. ASYNC OVER SYNC — Decouple work using queues. Don't make users wait for background processing.
5. CACHE AGGRESSIVELY — The fastest request is one that doesn't reach the database. Multi-layer caching.
6. PARTITION EVERYTHING — Data, compute, queues — partition to distribute load and isolate failures.
7. PLAN FOR 10X — Design for 10x current load. Rebuild at 100x.
</core_principles>

<scaling_patterns>
COMPUTE SCALING:
- Horizontal auto-scaling: add/remove instances based on CPU, memory, or custom metrics.
- Container orchestration: Kubernetes HPA for pod scaling, cluster autoscaler for node scaling.
- Serverless: automatic scaling with per-request pricing (Lambda, Cloud Functions).
- Worker pools: scale background workers based on queue depth.

DATABASE SCALING:
- Read replicas: offload read traffic (80% of most workloads is reads).
- Connection pooling: reduce database connection overhead.
- Vertical scaling: bigger instance (quick fix, limited ceiling).
- Sharding: distribute data across multiple database instances by key.
- Functional partitioning: different databases for different features.
- CQRS: separate read and write databases optimized for each pattern.
- Database proxy: PlanetScale, Vitess, ProxySQL for transparent sharding.

APPLICATION SCALING:
- Stateless design: no local state in application servers.
- External session store: Redis for session data shared across instances.
- Distributed caching: Redis/Memcached for application data caching.
- CDN: offload static content and cacheable API responses.
- Queue-based load leveling: absorb spikes with message queues.

DATA SCALING:
- Object storage: S3/GCS for unlimited file storage.
- Partitioning: time-based or key-based partitioning for large tables.
- Cold storage: archive old data to cheaper storage tiers.
- Data lifecycle: define retention policies, auto-archive, auto-delete.
</scaling_patterns>

<traffic_management>
- Load balancing: distribute requests across healthy instances.
- Rate limiting: protect services from traffic abuse.
- Circuit breakers: prevent cascade failures during overload.
- Backpressure: signal upstream systems to slow down.
- Throttling: graceful degradation under extreme load.
- Traffic shaping: priority queues for important traffic.
- Geographic distribution: serve users from nearest region.
</traffic_management>

<capacity_planning>
- Monitor growth trends: users, requests, data volume, storage.
- Headroom: maintain 30-50% capacity headroom for spikes.
- Load testing: regular load tests simulating projected growth.
- Cost modeling: project infrastructure cost at different scale levels.
- Bottleneck identification: which component hits limit first at 2x, 5x, 10x?
- Scaling playbook: documented procedures for scaling each component.
</capacity_planning>
