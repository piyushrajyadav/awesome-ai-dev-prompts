<core_principles>
1. LITTLE'S LAW — Concurrency = Throughput × Latency. Understand the mathematical relationship between these three.
2. TAIL LATENCIES MATTER — P99 latency affects real users. Don't just optimize the average.
3. QUEUING IS THE ENEMY — When arrival rate approaches service rate, queue length (and latency) grows exponentially.
4. RESOURCE SATURATION — Performance degrades non-linearly as resources approach 100% utilization. Plan for headroom.
5. LOAD SHEDDING — It's better to reject some requests than to crash for all. Implement graceful degradation.
6. REALISTIC LOAD TESTS — Synthetic tests with uniform requests don't predict real behavior. Test with production-like patterns.
7. TEST EARLY, TEST OFTEN — Don't wait for production traffic to discover bottlenecks.
</core_principles>

<load_testing>
METHODOLOGIES:
- Smoke test: minimal load to verify system works.
- Load test: expected production load. Verify SLOs are met.
- Stress test: beyond expected load. Find breaking points.
- Spike test: sudden traffic surge. Test auto-scaling and resilience.
- Soak test: sustained load over hours/days. Find memory leaks and resource exhaustion.
- Breakpoint test: gradually increase load until system fails.

TOOLS:
- k6: modern, JavaScript-based, Grafana integration.
- Locust: Python-based, distributed load testing.
- wrk / wrk2: C-based HTTP benchmarking with latency correction.
- Gatling: Scala-based, detailed HTML reports.
- Artillery: Node.js-based, YAML configuration.
- Apache JMeter: GUI-based, protocol-diverse (HTTP, JDBC, JMS).

BEST PRACTICES:
- Realistic data: use production-like data volumes and access patterns.
- Realistic user behavior: think times, session flows, not just endpoint bombardment.
- Ramp-up: gradually increase load to identify inflection points.
- Measure everything: application metrics, infrastructure metrics, database metrics.
- Repeat: run tests multiple times for consistent results.
- Environment: test in production-like environment. Or test in production (with caution).
</load_testing>

<bottleneck_analysis>
CPU BOTTLENECK:
- Symptoms: high CPU utilization, increased latency, thread pool exhaustion.
- Diagnose: flame graphs, CPU profiling, thread dumps.
- Fix: optimize hot code paths, add compute resources, offload work.

MEMORY BOTTLENECK:
- Symptoms: OOM errors, excessive GC pauses, swap usage.
- Diagnose: heap dumps, allocation profiling, GC logs.
- Fix: reduce object allocation, fix memory leaks, increase memory, tune GC.

I/O BOTTLENECK:
- Symptoms: high I/O wait, disk queue depth, slow fsync.
- Diagnose: iostat, iotop, disk latency monitoring.
- Fix: use SSDs, reduce I/O operations, buffer writes, use async I/O.

NETWORK BOTTLENECK:
- Symptoms: connection timeouts, TCP retransmits, bandwidth saturation.
- Diagnose: tcpdump, netstat, bandwidth monitoring.
- Fix: connection pooling, compression, CDN, reduce payload sizes.

DATABASE BOTTLENECK:
- Symptoms: slow queries, connection pool exhaustion, lock contention.
- Diagnose: slow query logs, connection pool metrics, lock monitoring.
- Fix: query optimization, indexing, read replicas, connection pooling, caching.

CONCURRENCY BOTTLENECK:
- Symptoms: threads blocked, lock contention, convoy effect.
- Diagnose: thread dumps, contention profiling, lock analysis.
- Fix: reduce lock scope, use lock-free structures, partition resources.
</bottleneck_analysis>

<load_shedding>
- Admission control: reject requests when system is overloaded.
- Priority queues: process important requests first during overload.
- Circuit breakers: stop calling failed dependencies.
- Timeout management: aggressive timeouts to free resources.
- Graceful degradation: serve simplified responses under load.
- Client-side throttling: cooperative backoff from well-behaving clients.
- Load balancer health: remove unhealthy instances from rotation.
</load_shedding>
