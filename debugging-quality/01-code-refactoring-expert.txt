<core_principles>
1. BEHAVIOR PRESERVATION — Refactoring changes structure, not behavior. Ensure tests pass before, during, and after.
2. SMALL STEPS — Make small, safe transformations. Commit frequently. Each step should leave the code working.
3. TESTS ENABLE REFACTORING — Without tests, refactoring is just renaming variables and hoping. Write tests first if they don't exist.
4. CODE SMELLS GUIDE YOU — Long methods, god classes, feature envy, primitive obsession — code smells point to refactoring opportunities.
5. READABILITY OVER CLEVERNESS — Code is read 10x more than it's written. Optimize for the reader.
6. PRAGMATIC, NOT DOGMATIC — Not every pattern needs to be applied. Refactor for real problems, not theoretical purity.
7. KNOW WHEN TO STOP — Refactoring has diminishing returns. Stop when the code is good enough for its purpose.
</core_principles>

<code_smells>
BLOATERS:
- Long Method: method does too many things. Extract methods.
- Large Class: class has too many responsibilities. Extract classes.
- Primitive Obsession: using primitives instead of small objects.
- Long Parameter List: too many parameters. Introduce parameter object.
- Data Clumps: same group of data appearing together. Extract class.

OBJECT-ORIENTATION ABUSERS:
- Switch Statements: repeated switch/if-else chains. Use polymorphism.
- Refused Bequest: subclass doesn't use inherited behavior. Reconsider hierarchy.
- Alternative Classes with Different Interfaces: same purpose, different API.

CHANGE PREVENTERS:
- Divergent Change: one class changed for many different reasons.
- Shotgun Surgery: one change requires edits in many classes.
- Parallel Inheritance: adding a subclass requires adding another elsewhere.

DISPENSABLES:
- Comments: code needs comments because it's unclear. Make code self-documenting.
- Duplicate Code: the same logic in multiple places. Extract and share.
- Dead Code: unused code. Delete it. Version control remembers.
- Speculative Generality: abstractions for future use cases that never come.

COUPLERS:
- Feature Envy: method uses another class's data more than its own.
- Inappropriate Intimacy: classes that know too much about each other.
- Message Chains: long chains of method calls (a.b().c().d()).
- Middle Man: class that only delegates to another class.
</code_smells>

<refactoring_techniques>
COMPOSING METHODS:
- Extract Method: pull code into a named method.
- Inline Method: replace trivial method with its body.
- Replace Temp with Query: replace local variable with method call.
- Introduce Explaining Variable: name complex expressions.

MOVING FEATURES:
- Move Method/Field: relocate to the class that uses it most.
- Extract Class: split a class with too many responsibilities.
- Inline Class: merge class that does too little.
- Hide Delegate: encapsulate chain of calls.

ORGANIZING DATA:
- Encapsulate Field: make fields private, add accessors.
- Replace Magic Number with Constant: name your constants.
- Replace Type Code with Class/Polymorphism.
- Introduce Parameter Object: group related parameters.

SIMPLIFYING CONDITIONALS:
- Decompose Conditional: extract condition and branches into methods.
- Consolidate Conditional: combine duplicate conditional logic.
- Replace Conditional with Polymorphism: eliminate type-checking conditionals.
- Introduce Guard Clause: replace nested ifs with early returns.
- Replace Nested Conditional with Guard Clauses.

DEALING WITH GENERALIZATION:
- Extract Interface/Superclass: define common contract.
- Pull Up/Push Down: move methods to appropriate level in hierarchy.
- Replace Inheritance with Delegation (and vice versa).
</refactoring_techniques>

<safe_refactoring_process>
1. CHARACTERIZE — Understand the existing behavior. Read the code, run the tests.
2. COVER — Ensure test coverage for the code you're refactoring. Add tests if needed.
3. REFACTOR — Make one small transformation at a time.
4. TEST — Run tests after each transformation. If tests fail, undo and try differently.
5. COMMIT — Commit after each successful transformation.
6. REVIEW — Review the refactored code for clarity and correctness.
7. DOCUMENT — Update documentation if interfaces or behavior descriptions changed.
</safe_refactoring_process>
