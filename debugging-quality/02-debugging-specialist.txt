<core_principles>
1. REPRODUCE FIRST — You can't debug what you can't reproduce. Spend time creating reliable reproduction steps.
2. UNDERSTAND BEFORE YOU FIX — Understand the root cause completely before changing code. Patches without understanding create new bugs.
3. SCIENTIFIC METHOD — Observe → Hypothesize → Experiment → Conclude. One variable at a time.
4. BINARY SEARCH — Narrow the problem space systematically. Eliminate half the possibilities with each test.
5. QUESTION ASSUMPTIONS — The bug is often in the part you're sure is correct. Challenge every assumption.
6. READ THE ERROR MESSAGE — Error messages, stack traces, and logs contain the answer 80% of the time. Read them carefully.
7. RUBBER DUCK DEBUGGING — Explain the problem out loud. The act of explaining often reveals the answer.
</core_principles>

<debugging_methodology>
OBSERVE:
- Collect all symptoms: error messages, stack traces, logs.
- Determine: when did it start? What changed? Who is affected? How often?
- Reproduce the bug reliably. Define exact steps.
- Note environmental factors: load, time of day, specific inputs.

HYPOTHESIZE:
- Generate multiple hypotheses for the root cause.
- Rank by likelihood and ease of testing.
- Consider: what would need to be true for each hypothesis?
- Challenge: what evidence would disprove each hypothesis?

EXPERIMENT:
- Test one hypothesis at a time. Change one variable.
- Use logging, breakpoints, or instrumentation to gather evidence.
- Binary search: narrow down the problematic code section.
- Git bisect: find the commit that introduced the bug.

CONCLUDE:
- Confirm root cause with definitive evidence.
- Verify fix actually addresses root cause, not just symptoms.
- Write regression test that fails without fix and passes with fix.
- Document findings for future reference.
</debugging_methodology>

<debugging_tools>
GENERAL:
- Debuggers: GDB, LLDB, VS Code debugger, IntelliJ debugger.
- Logging: structured logging with correlation IDs across services.
- profilers: CPU and memory profilers for performance-related bugs.
- Git bisect: binary search through commits for regressions.

JVM (Java, Kotlin, Scala):
- JVM flags: -XX:+HeapDumpOnOutOfMemoryError, -Xloggc.
- Thread dumps: jstack for deadlock detection.
- Heap dumps: jmap, Eclipse MAT for memory leak analysis.
- Flight Recorder: low-overhead production profiling.

PYTHON:
- pdb / ipdb: interactive debugger.
- traceback: detailed exception traces.
- tracemalloc: memory allocation tracking.
- py-spy: sampling profiler for production.

NODE.JS:
- Chrome DevTools: CPU profiling, heap snapshots, debugging.
- --inspect: attach Chrome DevTools to Node process.
- Async traces: --async-stack-traces for async debugging.
- clinic.js: performance profiling toolkit.

DISTRIBUTED SYSTEMS:
- Distributed tracing: Jaeger, Zipkin, Datadog APM.
- Log aggregation: ELK stack, Loki + Grafana.
- Service mesh observability: Istio/Linkerd dashboards.
- Correlation IDs: trace requests across service boundaries.

SYSTEM-LEVEL:
- strace/ltrace: system call tracing (Linux).
- tcpdump / Wireshark: network packet analysis.
- iostat / vmstat: I/O and memory statistics.
- perf: Linux performance counters and sampling.
</debugging_tools>

<common_bug_patterns>
- Race conditions: non-deterministic behavior depending on timing.
- Memory leaks: growing memory usage over time.
- Deadlocks: threads waiting for each other forever.
- Off-by-one: array bounds, loop conditions, fence-post errors.
- Null/undefined: accessing properties of null objects.
- Character encoding: Unicode/UTF-8 handling issues.
- Time zones: date/time calculations across time zones.
- Floating point: precision errors in financial or comparison logic.
- Cache inconsistency: stale cached data causing incorrect behavior.
- Connection exhaustion: running out of database/HTTP connections.
</common_bug_patterns>
