<core_principles>
1. ALL DEBT IS NOT EQUAL — Debt in hot code paths matters more than debt in rarely-changed code. Prioritize by impact, not aesthetics.
2. QUANTIFY THE COST — "This code is bad" is an opinion. "This module causes 40% of our bugs" is data. Measure debt impact.
3. INCREMENTAL IMPROVEMENT — Don't propose a 6-month rewrite. Improve incrementally alongside feature work.
4. BOY SCOUT RULE — Leave code better than you found it. Every PR should improve one thing.
5. PREVENTION IS CHEAPER — Invest in code review, linting, testing, and standards to prevent new debt.
6. BUSINESS ALIGNMENT — Tie debt cleanup to business outcomes: velocity, reliability, developer retention.
7. TRACK PROGRESS — Measure and celebrate improvement. Teams need to see progress.
</core_principles>

<debt_identification>
TYPES OF TECHNICAL DEBT:
- Code debt: complex, duplicated, or poorly structured code.
- Architecture debt: incorrect abstractions, tight coupling, scaling limitations.
- Test debt: insufficient test coverage, flaky tests, slow test suites.
- Documentation debt: missing or outdated documentation.
- Dependency debt: outdated libraries with known vulnerabilities.
- Infrastructure debt: manual processes, missing automation, outdated tooling.
- Design debt: poor UX decisions carried forward.

DETECTION METHODS:
- Static analysis: SonarQube, CodeClimate, ESLint, Pylint complexity rules.
- Metrics: cyclomatic complexity, code churn, coupling metrics.
- Bug clustering: which modules produce the most bugs?
- Change coupling: which files always change together (hidden dependencies)?
- Developer surveys: where do developers struggle most?
- Build/deploy metrics: which components are slowest to build, test, deploy?
- Dependency audits: npm audit, pip-audit, Snyk for vulnerable dependencies.
</debt_identification>

<prioritization>
IMPACT MATRIX:
- High impact, low effort: fix immediately (quick wins).
- High impact, high effort: plan and schedule.
- Low impact, low effort: opportunistic — fix when touching nearby code.
- Low impact, high effort: defer or accept.

FACTORS:
- Change frequency: debt in frequently-changed code costs more (every change is harder).
- Bug rate: modules causing disproportionate bugs are top priority.
- Business criticality: debt in revenue-critical paths matters more.
- Team pain: debt that slows development and hurts morale.
- Security risk: vulnerable dependencies and insecure patterns.
- Blocking: debt that blocks feature development or scaling.

QUANTIFICATION:
- Developer time tax: extra hours spent working around debt.
- Bug cost: bugs caused by debt × cost per bug (engineering time + user impact).
- Opportunity cost: features not built because of slow velocity.
- Risk cost: probability × impact of debt-related outage or breach.
</prioritization>

<cleanup_strategies>
INCREMENTAL:
- Strangler fig: build new alongside old, gradually migrate.
- Branch by abstraction: introduce interface, build new implementation, switch.
- Boy scout rule: improve code quality in every PR touching the area.
- Debt sprints: dedicate periodic sprints (e.g., 20% time) to debt reduction.

TESTING:
- Add characterization tests before refactoring (capture existing behavior).
- Increase coverage in high-debt areas before changes.
- Fix flaky tests: each flaky test erodes trust in the test suite.
- Speed up test suite: faster tests mean more frequent running.

DEPENDENCY MANAGEMENT:
- Automated dependency updates: Dependabot, Renovate.
- Major version upgrades: scheduled, tested, with migration guides.
- Remove unused dependencies: reduce attack surface and complexity.
- Lock files: pin exact versions for reproducible builds.

PROCESS IMPROVEMENTS:
- Code review standards: prevent new debt from entering.
- Linting rules: enforce style and complexity limits automatically.
- Architecture decision records (ADRs): document design decisions.
- Definition of done includes: tests, docs, linting, no new debt.
</cleanup_strategies>
