<core_principles>
1. TESTING PYRAMID — Many fast unit tests, fewer integration tests, fewest e2e tests. Invert the pyramid at your peril.
2. TEST BEHAVIOR, NOT IMPLEMENTATION — Tests should verify what the code does, not how it does it. Implementation changes shouldn't break tests.
3. FAST FEEDBACK — Tests must run fast. Slow tests don't get run. Aim for <10 minute CI.
4. INDEPENDENT TESTS — Each test should pass or fail independently. No test should depend on another test's state.
5. READABLE TESTS — Tests are documentation. Anyone should understand what's being tested and why.
6. DETERMINISTIC — Tests must produce the same result every run. Flaky tests destroy trust.
7. MAINTAINABLE — Test code is code. Apply the same quality standards: DRY, clear naming, good structure.
</core_principles>

<testing_pyramid>
UNIT TESTS:
- Test individual functions/classes in isolation.
- Fast: milliseconds per test. Run thousands in seconds.
- Mock external dependencies (database, APIs, file system).
- High coverage: 80%+ line coverage for business logic.
- Frameworks: Jest, pytest, JUnit, Go testing, xUnit.

INTEGRATION TESTS:
- Test interactions between components (service + database, API + external service).
- Use real dependencies: test database (Docker), test APIs (WireMock).
- Moderate speed: seconds per test.
- Focus on: database queries, API contracts, message handling.
- Frameworks: Testcontainers, SuperTest, Spring Boot Test.

END-TO-END TESTS:
- Test complete user flows through the entire system.
- Slowest and most fragile. Keep the number small.
- Focus on critical user journeys only.
- Frameworks: Playwright, Cypress, Selenium.
- Run in CI after deployment to a test environment.

CONTRACT TESTS:
- Verify API compatibility between services.
- Consumer-driven: consumers define expected API behavior.
- Faster than e2e, catches integration issues earlier.
- Framework: Pact.

PROPERTY-BASED TESTS:
- Generate random inputs to find edge cases automatically.
- Define properties that should always be true.
- Finds bugs that example-based tests miss.
- Frameworks: Hypothesis (Python), fast-check (JS), QuickCheck.
</testing_pyramid>

<test_design_patterns>
ARRANGE-ACT-ASSERT (AAA):
- Arrange: set up test data and dependencies.
- Act: execute the code under test.
- Assert: verify the expected outcome.
- One concept per test. Clear separation of phases.

TEST FIXTURES:
- Factory functions: create test data with sensible defaults.
- Builder pattern: fluent test data construction.
- Fixtures: reusable setup/teardown (pytest fixtures, JUnit @BeforeEach).
- Database seeding: consistent test data for integration tests.

MOCKING:
- Stubs: return predetermined values.
- Mocks: verify interactions (called with expected arguments).
- Fakes: simplified working implementations (in-memory database).
- Spy: record calls for later assertion.
- Don't over-mock: too many mocks means you're testing the mocks, not the code.

TEST DATA:
- Factories over fixtures: generate test data programmatically.
- Minimal data: use only the data needed for each test.
- Isolated data: each test creates its own data. No shared state.
- Anonymize production data for realistic test scenarios.
</test_design_patterns>

<test_quality>
WHAT MAKES A GOOD TEST:
- Fast: runs in milliseconds (unit) or seconds (integration).
- Isolated: no dependencies on other tests or shared state.
- Repeatable: same result every time, on any machine.
- Self-validating: passes or fails without human interpretation.
- Timely: written close to the code it tests.

FIGHTING FLAKY TESTS:
- Identify: track test results over time, flag inconsistent tests.
- Quarantine: move flaky tests to a separate suite. Don't ignore them.
- Root causes: timing dependencies, shared state, network calls, random data.
- Fix: use retries cautiously (for external deps only), fix root causes.

COVERAGE:
- Line coverage: percentage of lines executed by tests.
- Branch coverage: percentage of conditional branches tested.
- Mutation testing: verify tests actually detect code changes (PIT, Stryker).
- Coverage targets: 80%+ for business logic, don't chase 100% blindly.
</test_quality>
