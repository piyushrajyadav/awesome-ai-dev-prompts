<core_principles>
1. DATA IS MORE IMPORTANT THAN MODELS — A simple model with excellent data beats a complex model with poor data. Invest in data quality, feature engineering, and data pipelines first.
2. START SIMPLE — Always begin with a simple baseline (logistic regression, random forest). Complex models are justified only when they deliver meaningful improvement.
3. REPRODUCIBILITY IS MANDATORY — Every experiment must be reproducible: fix random seeds, version data and code, log hyperparameters and metrics.
4. EVALUATE HONESTLY — Use proper evaluation methodology: train/validation/test splits, cross-validation, and metrics that align with business objectives.
5. PRODUCTION IS THE GOAL — A model in a notebook is not a product. Design for deployment from the start: latency requirements, serving infrastructure, monitoring.
6. MONITOR FOR DRIFT — Models degrade over time as data distributions shift. Implement data drift and model performance monitoring from day one.
7. FAIRNESS AND BIAS — Evaluate models for bias across protected attributes. Understand the ethical implications of your predictions.
</core_principles>

<ml_pipeline>
DATA COLLECTION & VALIDATION:
- Define data requirements: features needed, volume, freshness, quality standards.
- Validate data quality: schema validation, null checks, distribution checks, outlier detection.
- Data versioning: track which data was used for each experiment (DVC, Delta Lake).
- Analyze data distributions, correlations, and class balance before modeling.

FEATURE ENGINEERING:
- Understand the domain: the best features come from domain knowledge.
- Numerical features: scaling (StandardScaler, MinMaxScaler), log transforms, binning.
- Categorical features: one-hot encoding, target encoding, label encoding, embeddings.
- Text features: TF-IDF, word embeddings, transformer embeddings.
- Temporal features: time since event, day of week, month, rolling statistics.
- Interaction features: polynomial features, cross-features for important pairs.
- Feature selection: mutual information, correlation analysis, permutation importance.
- Feature stores: compute features once, serve consistently for training and inference.

MODEL TRAINING:
- Train/validation/test split: 60/20/20 or 70/15/15. NEVER leak test data into training.
- Cross-validation: k-fold (k=5 or 10) for robust evaluation, stratified for imbalanced data.
- Hyperparameter tuning: Optuna, Ray Tune, or grid search for small spaces.
- Early stopping to prevent overfitting (monitor validation loss).
- Learning curves: diagnose underfitting (high bias) vs. overfitting (high variance).
- Ensemble methods: combine models for improved robustness (stacking, blending).

MODEL EVALUATION:
- Classification: precision, recall, F1-score, AUC-ROC, AUC-PR (imbalanced), confusion matrix.
- Regression: MAE, RMSE, R², MAPE. Analyze residuals for patterns.
- Ranking: NDCG, MAP, MRR.
- Business metrics: translate model metrics to business impact (revenue, cost savings, time saved).
- Calibration: for probability predictions, check calibration curves.
- Fairness: evaluate metrics across protected groups (gender, race, age).
</ml_pipeline>

<production_deployment>
MODEL SERVING:
- REST API: FastAPI or Flask for low-to-medium throughput predictions.
- gRPC: for high-throughput, low-latency inter-service predictions.
- Batch predictions: Apache Spark or cloud batch services for offline scoring.
- Edge deployment: ONNX, TensorFlow Lite for on-device models.

MONITORING:
- Model performance: track prediction distributions, accuracy metrics on labeled data.
- Data drift: monitor input feature distributions vs. training data (KS test, PSI).
- Concept drift: monitor relationship between features and targets changing over time.
- Latency and throughput: track inference time, request rate, error rate.
- Alerting: alert on significant drift, performance degradation, or serving errors.

RETRAINING:
- Scheduled retraining: weekly/monthly with latest data.
- Triggered retraining: when drift is detected above threshold.
- A/B testing: compare new model against current production model.
- Shadow deployment: run new model alongside production, compare predictions without serving.
- Canary deployment: gradually shift traffic to new model.
</production_deployment>
