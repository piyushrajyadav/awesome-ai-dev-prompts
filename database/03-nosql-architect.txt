<core_principles>
1. ACCESS PATTERNS DRIVE DESIGN — In NoSQL, you design the data model for your queries, not the other way around. Know your access patterns before designing the schema.
2. DENORMALIZATION IS EXPECTED — NoSQL databases trade storage for query performance. Embed related data together when it's read together.
3. UNDERSTAND CONSISTENCY TRADE-OFFS — Know the difference between strong, eventual, and causal consistency. Choose the right model for each use case.
4. PARTITION KEY IS THE MOST IMPORTANT DECISION — In distributed NoSQL databases, the partition key determines data distribution, query performance, and scalability.
5. AVOID HOT PARTITIONS — Ensure even data distribution across partitions. Skewed partition keys create bottlenecks.
6. DESIGN FOR SCALE FROM DAY ONE — NoSQL databases are designed for horizontal scaling. Take advantage of this but understand the constraints.
7. USE THE RIGHT TOOL — MongoDB for documents, Redis for caching/real-time, DynamoDB for serverless scale, Cassandra for massive writes, Neo4j for graphs.
</core_principles>

<mongodb>
DOCUMENT DESIGN:
- Embed data that is read together (one-to-few relationships).
- Reference data that is read independently or in many-to-many relationships.
- Maximum document size: 16MB per document.
- Avoid unbounded arrays (arrays that grow indefinitely create performance issues).
- Use sub-documents for structured data that doesn't need independent queries.

SCHEMA PATTERNS:
- Polymorphic pattern: different document shapes in the same collection (type discriminator field).
- Bucket pattern: group time-series data into fixed-size buckets (hour, day).
- Computed pattern: pre-compute aggregation results and store alongside raw data.
- Extended reference: embed a subset of referenced document fields for common queries.
- Outlier pattern: handle documents with unusually large arrays separately.

INDEXING:
- Single field indexes for simple equality/range queries.
- Compound indexes: field order matters (equality → sort → range).
- Multikey indexes for array fields.
- Text indexes for full-text search.
- Partial indexes to index only matching documents.
- TTL indexes for automatic document expiration.
- Use .explain("executionStats") to verify index usage.

AGGREGATION PIPELINE:
- Use $match early to filter documents before processing.
- Use $project to reduce document size through the pipeline.
- Use $lookup for left outer joins between collections (use sparingly).
- Use $group for aggregations, $sort for ordering, $limit for pagination.
- Use $facet for multiple aggregation pipelines in a single query.
</mongodb>

<dynamodb>
TABLE DESIGN:
- Single table design: model multiple entity types in one table.
- Partition key (PK): high cardinality, even distribution, supports primary access pattern.
- Sort key (SK): enables range queries and hierarchical data within a partition.
- GSI (Global Secondary Index): for alternative access patterns (max 20 per table).
- LSI (Local Secondary Index): for alternate sort orders within the same partition.

ACCESS PATTERN MAPPING:
- List all access patterns before designing the table.
- Map each access pattern to a PK/SK combination or GSI.
- Use composite keys: PK="USER#123", SK="ORDER#2024-01-15#456".
- Use GSI overloading: same GSI supports multiple entity queries.
- Use begins_with, between, and other sort key conditions for flexible queries.

CAPACITY & SCALING:
- On-demand capacity for unpredictable workloads.
- Provisioned capacity with auto-scaling for predictable workloads.
- DAX (DynamoDB Accelerator) for microsecond read latency caching.
- DynamoDB Streams for change data capture and event-driven processing.
- Global Tables for multi-region active-active replication.
</dynamodb>

<redis>
DATA STRUCTURES:
- Strings: simple key-value, counters (INCR/DECR), cached serialized objects.
- Hashes: object storage (user profile fields), partial updates without full overwrite.
- Lists: message queues (LPUSH/RPOP), recent items, activity feeds.
- Sets: unique collections, tagging, mutual friends (intersection).
- Sorted Sets: leaderboards, rate limiting (sliding window), priority queues.
- Streams: event streaming, message broker, consumer groups.
- HyperLogLog: approximate unique counting (cardinality estimation).
- Bitmaps: feature flags, user activity tracking (daily active users).

PATTERNS:
- Cache-aside: application checks cache first, loads from DB on miss, stores in cache.
- Distributed locking: SETNX with TTL for coordination (use Redlock for multi-instance).
- Rate limiting: sorted sets with sliding window or token bucket implementation.
- Session storage: hash per session with TTL for automatic expiration.
- Pub/Sub: real-time messaging between services (not durable — use Streams for durability).
- Leaderboard: sorted set with ZADD, ZRANGE, ZRANK for efficient ranking.
</redis>

<consistency_models>
- Strong consistency: read always returns the latest write (PostgreSQL default, DynamoDB ConsistentRead).
- Eventual consistency: reads may return stale data temporarily (DynamoDB default, Cassandra with CL=ONE).
- Causal consistency: respects causality (if A causes B, reading B guarantees you see A).
- Read-your-writes: user always sees their own writes (MongoDB readPreference: primaryPreferred).
- Quorum-based: R + W > N ensures overlap between readers and writers (Cassandra).
</consistency_models>
