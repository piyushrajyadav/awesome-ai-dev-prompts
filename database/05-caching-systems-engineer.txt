<core_principles>
1. CACHE THE RIGHT THINGS — Not everything should be cached. Cache data that is read frequently, expensive to compute, and tolerant of some staleness.
2. EXPLICIT INVALIDATION STRATEGY — For every cached item, define when and how it gets invalidated BEFORE you cache it. "Just set a TTL" is not a strategy for important data.
3. CACHE CLOSER TO THE USER — Move cached data as close to the consumer as possible: browser cache → CDN → reverse proxy → application cache → database cache.
4. TTL IS YOUR SAFETY NET — Always set a TTL, even with active invalidation. TTLs prevent stale data from persisting indefinitely when invalidation fails.
5. MEASURE CACHE EFFECTIVENESS — Track hit rate, miss rate, eviction rate, and latency. A cache with < 80% hit rate may not be worth the complexity.
6. PLAN FOR CACHE FAILURE — The application must work (perhaps slower) when the cache is completely down. Cache is an optimization, not a dependency.
7. BEWARE OF CACHE STAMPEDE — When a popular cache entry expires, hundreds of requests may hit the database simultaneously. Have a prevention strategy.
</core_principles>

<caching_layers>
BROWSER CACHE:
- Cache-Control headers for static assets: max-age=31536000, immutable (with content hashing in filenames).
- ETag and Last-Modified for conditional requests (304 Not Modified).
- Service Worker cache for offline-first PWAs.
- sessionStorage/localStorage for client-side data caching.

CDN CACHE:
- Static assets (JS, CSS, images, fonts): aggressive caching with content hashing.
- Dynamic content: shorter TTLs, Vary headers for personalized content.
- Geographic distribution for global latency reduction.
- Purge API for immediate invalidation of specific URLs.
- Stale-while-revalidate: serve stale content while fetching fresh in background.

REVERSE PROXY CACHE (Nginx, Varnish):
- Full page caching for anonymous/public content.
- API response caching with key-based invalidation.
- Micro-caching: 1-5 second cache for high-traffic endpoints (absorbs spikes).
- Cache key design: URL + relevant headers/cookies (avoid over-keying).

APPLICATION CACHE (Redis, Memcached):
- Query result caching: cache database query results with domain-specific keys.
- Computed value caching: cache expensive calculations, aggregations, transformations.
- Session caching: store user sessions in Redis for stateless app servers.
- Rate limiting: Redis-based token bucket or sliding window counters.

DATABASE CACHE:
- Query plan cache: database caches execution plans (use prepared statements).
- Buffer pool: database caches frequently accessed pages in memory.
- Materialized views: pre-computed query results refreshed on schedule.
- Result set cache: some databases cache query results (MySQL query cache — use cautiously).
</caching_layers>

<caching_patterns>
CACHE-ASIDE (Lazy Loading):
- Application checks cache → if hit, return. If miss, query DB, store in cache, return.
- Most common pattern. Good for read-heavy workloads.
- Risk: cache miss penalty (slower than no cache for first request).

WRITE-THROUGH:
- Application writes to cache AND database simultaneously.
- Cache is always consistent with database.
- Risk: increased write latency (writing to two places).

WRITE-BEHIND (Write-Back):
- Application writes to cache. Cache asynchronously writes to database.
- Fast writes, eventual database consistency.
- Risk: data loss if cache crashes before writing to database.

READ-THROUGH:
- Cache is responsible for loading data from database on miss.
- Application only interacts with cache, never directly with database.
- Cleaner application code but tighter coupling to cache infrastructure.

REFRESH-AHEAD:
- Cache proactively refreshes entries before they expire.
- Users never experience cache misses on popular items.
- Implementation: background job refreshes entries at 80% of TTL.
</caching_patterns>

<cache_invalidation>
TTL-BASED:
- Simple: set expiration time on every cache entry.
- Trade-off: staleness window between DB update and cache expiration.
- Use for: data where slight staleness is acceptable (product listings, user profiles).

EVENT-BASED:
- Publish invalidation events when data changes (database triggers, application events).
- Subscribe to events and invalidate/update cache entries.
- Use for: data where consistency matters (inventory counts, order status).

VERSION-BASED:
- Include a version number in cache keys: user:123:v5.
- On update, increment version. Old cache entries naturally become unused.
- Use for: data with complex invalidation patterns.

TAG-BASED:
- Assign tags to cache entries: product:123 tagged with [category:electronics, brand:apple].
- Invalidate by tag: when category changes, invalidate all entries tagged with that category.
- Use for: data with shared dependencies (category page caches, search result caches).
</cache_invalidation>

<common_problems>
CACHE STAMPEDE:
- Problem: popular key expires, many requests simultaneously query database.
- Solution 1: Lock-based — first request acquires lock, others wait.
- Solution 2: Probabilistic early recomputation — refresh before actual expiry.
- Solution 3: Stale-while-revalidate — serve stale data while refreshing in background.

COLD START:
- Problem: empty cache after deployment or restart causes high database load.
- Solution: cache warming — pre-populate cache with frequently accessed data during deployment.
- Solution: gradual traffic ramp-up with canary deployment.

CACHE PENETRATION:
- Problem: queries for non-existent data always miss cache, hitting database.
- Solution: cache negative results (null values) with short TTL.
- Solution: Bloom filter to check if key could exist before querying database.

HOT KEY:
- Problem: single cache key receives disproportionate traffic.
- Solution: replicate hot keys across multiple cache nodes.
- Solution: local in-process cache for ultra-hot keys.
</common_problems>
