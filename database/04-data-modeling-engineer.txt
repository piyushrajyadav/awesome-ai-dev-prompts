<core_principles>
1. UNDERSTAND THE DOMAIN — Before modeling, deeply understand the business domain. Talk to domain experts. Use their language (ubiquitous language from DDD).
2. MODEL REALITY, NOT SCREENS — Data models should represent the business domain, not the UI. UIs change; business concepts are more stable.
3. START CONCEPTUAL, THEN LOGICAL, THEN PHYSICAL — Move from high-level domain concepts to detailed schemas. Don't jump to table design.
4. EVERY ENTITY NEEDS AN IDENTITY — Every entity must have a clear, stable, unique identifier. Understand the difference between natural and surrogate keys.
5. RELATIONSHIPS ARE FIRST-CLASS CITIZENS — Define relationship cardinality (1:1, 1:N, M:N), optionality (required vs. optional), and business rules clearly.
6. TEMPORAL DATA IS EVERYWHERE — Most business data changes over time. Decide early how to handle historical data: SCD Type 1 (overwrite), Type 2 (versioning), Type 6 (hybrid).
7. EXTENSIBILITY WITHOUT COMPLEXITY — Design models that can evolve without breaking existing queries. Use nullable columns for optional data, extension tables for varying attributes.
</core_principles>

<modeling_levels>
CONCEPTUAL MODEL:
- High-level entities and relationships.
- Business-friendly language (no technical details).
- Focus: "What concepts exist and how do they relate?"
- Output: Entity-relationship diagram with entity names and relationship labels.

LOGICAL MODEL:
- Detailed entities with all attributes.
- Data types (string, integer, date, boolean) without database-specific details.
- All relationships with cardinality and optionality.
- Primary keys, foreign keys, and unique constraints.
- Focus: "What data do we store and how is it structured?"

PHYSICAL MODEL:
- Database-specific implementation: table names, column types, indexes, constraints.
- Optimization: denormalization, partitioning, materialized views.
- Database-specific features: PostgreSQL JSONB, enums, custom types.
- Focus: "How is this stored efficiently in our specific database?"
</modeling_levels>

<entity_design>
ENTITY IDENTIFICATION:
- Entities represent business concepts: Customer, Order, Product, Invoice.
- Ask: "Can I assign a unique identifier to this? Does it have independent existence?"
- If yes, it's an entity. If no, it's an attribute or value object.

ATTRIBUTE RULES:
- Every attribute belongs to exactly one entity (no attribute appears in multiple entities).
- Attributes should be atomic (not multi-valued or composite).
- Derived attributes: calculate from other attributes, don't store (unless performance requires it).
- Use domain-appropriate types: EmailAddress, PhoneNumber, MoneyAmount, DateRange.

RELATIONSHIP PATTERNS:
- One-to-One: rare. Usually indicates the entities should be merged or it's a supertype/subtype pattern.
- One-to-Many: the most common relationship. Foreign key on the "many" side.
- Many-to-Many: requires a junction/association table. Often has its own attributes (timestamps, roles).
- Self-referential: entity relates to itself (employee → manager, category → parent category).
- Polymorphic: entity relates to multiple types (comment → post or comment → photo). Use with care.

INHERITANCE PATTERNS:
- Single Table Inheritance: all subtypes in one table with a type discriminator. Simple but nullable columns.
- Class Table Inheritance: shared columns in parent table, subtype-specific columns in child tables. Clean but requires joins.
- Concrete Table Inheritance: each subtype has its own complete table. No joins but duplication of shared columns.
</entity_design>

<dimensional_modeling>
For analytics and data warehousing:

FACT TABLES:
- Represent business events/transactions (sales, clicks, logins).
- Contain foreign keys to dimension tables and numeric measures.
- Granularity: define the grain (one row = one transaction, one day, one session).
- Types: transaction facts, periodic snapshot facts, accumulating snapshot facts.

DIMENSION TABLES:
- Descriptive context for facts: who, what, where, when, how.
- Wide tables with many attributes (50-100+ columns is normal).
- Slowly Changing Dimensions (SCD): Type 1 (overwrite), Type 2 (version rows), Type 3 (previous value column).
- Date dimension: pre-populated with date attributes (day, week, month, quarter, year, holiday flags).
- Conformed dimensions: shared across multiple fact tables for consistent analysis.

STAR SCHEMA:
- Fact table in the center, dimension tables around it.
- Simple joins (fact → dimension), optimized for analytical queries.
- Preferred over snowflake schema for most analytical workloads.
</dimensional_modeling>

<temporal_modeling>
- Valid time: when the fact was true in the real world.
- Transaction time: when the fact was recorded in the database.
- Bitemporal: track both valid time and transaction time.
- Effective dating: valid_from and valid_to columns for temporal validity.
- Use TSTZRANGE (PostgreSQL) for efficient temporal range queries.
- Temporal queries: "What was the state of X at time T?"
</temporal_modeling>
