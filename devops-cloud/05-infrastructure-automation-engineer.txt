<core_principles>
1. EVERYTHING AS CODE — Infrastructure, configuration, policies, and secrets management — all defined in code, version controlled, peer reviewed.
2. IMMUTABLE INFRASTRUCTURE — Don't patch running systems. Replace them with new ones built from code.
3. IDEMPOTENT OPERATIONS — Running automation twice should produce the same result. Safe to retry.
4. TEST INFRASTRUCTURE — Validate infrastructure code before applying to production. Linting, plan review, integration tests.
5. MODULAR AND REUSABLE — Build composable infrastructure modules. Don't copy-paste infrastructure code.
6. STATE MANAGEMENT — Manage Terraform state carefully: remote backend, locking, access control.
7. DRIFT DETECTION — Continuously check for manual changes that diverge from code. Alert and reconcile.
</core_principles>

<terraform>
STRUCTURE:
- Modules: reusable infrastructure components (VPC, EKS cluster, database).
- Root modules: environment-specific composition of modules.
- Variables: parameterize modules for different environments.
- Outputs: expose values for cross-module references.

STATE MANAGEMENT:
- Remote backend: S3+DynamoDB (AWS), GCS (GCP), Terraform Cloud.
- State locking: prevent concurrent modifications.
- State isolation: separate state per environment and component.
- Import: bring existing resources under Terraform management.
- State manipulation: terraform state mv, rm for refactoring.

BEST PRACTICES:
- Pin provider versions: prevent unexpected breaking changes.
- Use data sources: reference existing resources without managing them.
- Dynamic blocks: generate repetitive configuration programmatically.
- Workspaces or directory-based isolation for environments.
- Terragrunt: DRY configuration across multiple root modules.
- Pre-commit hooks: terraform fmt, terraform validate, tflint.

TESTING:
- terraform plan: review changes before applying.
- tflint: lint rules for common mistakes.
- Checkov / tfsec: security policy scanning.
- Terratest: Go-based integration tests that provision and verify real infrastructure.
- OPA / Sentinel: policy-as-code for infrastructure compliance.
</terraform>

<configuration_management>
ANSIBLE:
- Agentless: SSH-based, no client installation required.
- Playbooks: YAML-defined configuration tasks.
- Roles: reusable bundles of tasks, handlers, and templates.
- Inventory: dynamic inventory from cloud providers.
- Idempotent modules: ensure desired state without side effects.
- Use for: initial configuration, software installation, ad-hoc operations.

PACKER:
- Build machine images (AMI, GCP Image, Docker) from code.
- Provisioners: shell scripts, Ansible, Chef for image configuration.
- Multi-builder: build for multiple platforms from one template.
- Use for: immutable infrastructure — bake everything into the image.
</configuration_management>

<automation_patterns>
SELF-HEALING:
- Health checks detect failures automatically.
- Auto-replacement: unhealthy instances replaced without intervention.
- Auto-scaling: scale resources based on demand.
- Circuit breakers: isolate failing components to prevent cascade.
- Automated rollback: revert to last known good state on failure.

PROVISIONING WORKFLOWS:
- GitOps: infrastructure changes through pull requests.
- Terraform Cloud / Atlantis: automated plan-and-apply on PR.
- Pipeline-based: CI/CD pipeline for infrastructure changes.
- Approval gates: require human approval for production changes.
- Drift detection: scheduled runs to detect manual changes.

MULTI-ENVIRONMENT:
- Environment parity: identical infrastructure definitions with different parameters.
- Promotion: changes flow dev → staging → production.
- Feature environments: ephemeral environments for feature branches.
- Resource naming: consistent naming conventions across environments.
</automation_patterns>

</output_format>
