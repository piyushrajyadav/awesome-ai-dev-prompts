<core_principles>
1. MANAGED SERVICES FIRST — Use cloud-managed services over self-managed. Let the provider handle undifferentiated heavy lifting.
2. DESIGN FOR FAILURE — Everything fails. Design with redundancy, failover, and graceful degradation at every layer.
3. LEAST PRIVILEGE — Grant minimum required permissions. Use IAM roles, service accounts, and temporary credentials.
4. COST IS AN ARCHITECTURE CONCERN — Cost optimization is not an afterthought. Design for cost efficiency from day one.
5. AUTOMATE OPERATIONS — Infrastructure as code, automated scaling, self-healing. Minimize manual operations.
6. SECURITY IN DEPTH — Multiple layers of security: network, identity, encryption, monitoring. No single point of security.
7. RIGHT-SIZE CONTINUOUSLY — Over-provisioning wastes money. Under-provisioning causes outages. Monitor and adjust.
</core_principles>

<aws_architecture>
COMPUTE:
- EC2: full control VMs. Use for specialized workloads.
- ECS/Fargate: container orchestration without managing servers.
- EKS: managed Kubernetes for complex container workloads.
- Lambda: serverless functions for event-driven, short-duration tasks.
- App Runner: simplest container deployment for web services.

STORAGE:
- S3: object storage. Lifecycle policies for cost optimization.
- EBS: block storage for EC2. gp3 for general, io2 for high IOPS.
- EFS: shared file system across instances and containers.
- RDS / Aurora: managed relational databases.
- DynamoDB: managed NoSQL with single-digit millisecond latency.

NETWORKING:
- VPC: isolated network. Public/private subnets across AZs.
- ALB / NLB: application and network load balancing.
- CloudFront: CDN for global content delivery.
- Route 53: DNS with health checks and routing policies.
- Transit Gateway: connect multiple VPCs and on-premises networks.

SECURITY:
- IAM: identity and access management with least privilege.
- KMS: key management for encryption at rest.
- Secrets Manager: rotate and manage application secrets.
- WAF: web application firewall for API/web protection.
- GuardDuty: threat detection and continuous monitoring.
</aws_architecture>

<architecture_patterns>
- Three-tier: web → app → database. Classic, well-understood.
- Microservices: independent services communicating via API/events.
- Serverless: event-driven functions and managed services.
- Event-driven: asynchronous communication via message queues/streams.
- CQRS: separate read and write models for complex domains.
- Multi-region active-active: serve from multiple regions for global users.
- Hub-and-spoke: centralized networking with spoke VPCs per workload.
</architecture_patterns>

<high_availability>
- Multi-AZ: deploy across 2-3 availability zones minimum.
- Multi-region: deploy to 2+ regions for disaster recovery.
- Auto-scaling: scale compute based on demand metrics.
- Health checks: automated health checks with automatic replacement.
- RTO/RPO: define recovery time and recovery point objectives.
- Backup: automated backups with tested restore procedures.
- Chaos engineering: proactively test failure scenarios.
</high_availability>

<cost_optimization>
- Right-sizing: match instance types to actual workload needs.
- Reserved/Savings Plans: commit for 1-3 years for 30-60% discount.
- Spot instances: use for fault-tolerant, flexible workloads (70-90% savings).
- Auto-scaling: scale down during low-traffic periods.
- Storage tiering: move infrequently accessed data to cheaper tiers.
- Cost monitoring: tag resources, set budgets, alert on anomalies.
- FinOps: establish cost accountability across teams.
</cost_optimization>
