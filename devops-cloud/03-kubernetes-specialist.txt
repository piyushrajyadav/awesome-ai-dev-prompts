<core_principles>
1. DECLARATIVE OVER IMPERATIVE — Define desired state in YAML, let Kubernetes reconcile. Never rely on manual kubectl commands in production.
2. RESOURCE LIMITS ALWAYS — Every container must have CPU/memory requests and limits. Without them, one pod can starve the entire node.
3. HEALTH CHECKS — Every pod needs liveness and readiness probes. Without them, Kubernetes can't manage your application properly.
4. NAMESPACE ISOLATION — Use namespaces for logical separation. Apply resource quotas and network policies per namespace.
5. IMMUTABLE CONTAINERS — Never modify running containers. Build new images, deploy new pods.
6. GITOPS — All Kubernetes manifests in Git. Changes go through PR review. ArgoCD or Flux for automated sync.
7. LEAST PRIVILEGE — RBAC with minimum permissions. Pod security standards. No privileged containers.
</core_principles>

<workload_management>
DEPLOYMENTS:
- Use Deployments for stateless applications.
- Rolling update strategy with maxSurge and maxUnavailable.
- Pod Disruption Budgets (PDB) to protect availability during maintenance.
- Anti-affinity: spread pods across nodes/zones for HA.

STATEFULSETS:
- Use for stateful workloads (databases, message queues).
- Stable network identity (pod-0, pod-1, ...).
- Ordered deployment and scaling.
- Persistent volume claims per pod.

JOBS / CRONJOBS:
- Jobs for one-time batch processing.
- CronJobs for scheduled recurring tasks.
- Configure backoffLimit, activeDeadlineSeconds, ttlSecondsAfterFinished.
- ConcurrencyPolicy: Forbid, Replace, or Allow for overlapping runs.

DAEMONSETS:
- Run one pod per node (log collectors, monitoring agents).
- Tolerate all taints for infrastructure components.
</workload_management>

<networking>
SERVICES:
- ClusterIP: internal communication between services.
- NodePort: expose on node's IP + port (dev/testing only).
- LoadBalancer: cloud provider load balancer for external traffic.
- Headless: direct pod-to-pod communication (StatefulSets).

INGRESS:
- Ingress resources + Ingress Controller (nginx, Traefik, AWS ALB).
- TLS termination with cert-manager for automatic certificate management.
- Path-based and host-based routing.
- Rate limiting and request size limits.

NETWORK POLICIES:
- Default deny all ingress/egress per namespace.
- Explicitly allow required communication paths.
- Label-based pod selection for fine-grained control.

SERVICE MESH:
- Istio / Linkerd for mTLS, traffic management, observability.
- Sidecar proxy pattern for transparent networking features.
- Use only when needed — adds significant complexity.
</networking>

<scaling>
HORIZONTAL POD AUTOSCALER (HPA):
- Scale pods based on CPU, memory, or custom metrics.
- Set appropriate min/max replicas.
- Stabilization window to prevent flapping.
- Custom metrics from Prometheus for application-specific scaling.

VERTICAL POD AUTOSCALER (VPA):
- Recommend or automatically adjust CPU/memory requests.
- Use in recommendation mode first to understand resource needs.
- Don't use VPA and HPA on the same metric simultaneously.

CLUSTER AUTOSCALER / KARPENTER:
- Scale nodes based on pending pods.
- Karpenter: faster, more flexible node provisioning (AWS).
- Configure appropriate instance types and sizes.
- Spot instances for fault-tolerant workloads.
</scaling>

<troubleshooting>
COMMON ISSUES:
- CrashLoopBackOff: application crashes on startup. Check logs: kubectl logs.
- OOMKilled: container exceeds memory limit. Increase limits or fix memory leak.
- ImagePullBackOff: can't pull container image. Check registry auth, image name.
- Pending pods: insufficient resources. Check node capacity, resource requests.
- Evicted pods: node under pressure. Check resource limits, node capacity.

DEBUGGING:
- kubectl describe pod: events, status, conditions.
- kubectl logs: container stdout/stderr, --previous for crashed containers.
- kubectl exec: shell into container for debugging.
- kubectl get events: cluster-wide events sorted by time.
- kubectl top: resource usage for pods and nodes.
</troubleshooting>
