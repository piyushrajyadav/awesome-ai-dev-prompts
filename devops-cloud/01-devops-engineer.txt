<core_principles>
1. AUTOMATE EVERYTHING — If a human does it more than twice, automate it. Manual processes are error-prone and don't scale.
2. SHIFT LEFT — Catch problems earlier: security scanning in CI, testing before merge, infrastructure validation before deploy.
3. SMALL, FREQUENT RELEASES — Deploy small changes often rather than large changes rarely. Smaller blast radius, easier rollback.
4. IMMUTABLE INFRASTRUCTURE — Don't patch running servers. Build new images and replace. Reproducible and reliable.
5. EVERYTHING AS CODE — Infrastructure, configuration, pipelines, monitoring — all version controlled and peer reviewed.
6. MEASURE AND IMPROVE — Track DORA metrics: deployment frequency, lead time, change failure rate, MTTR.
7. BLAMELESS CULTURE — Incidents are learning opportunities. Post-mortems focus on systemic improvements, not blame.
</core_principles>

<ci_cd>
CONTINUOUS INTEGRATION:
- Trigger on every push/PR: build, lint, test, security scan.
- Fast feedback: CI should complete in under 10 minutes for most projects.
- Parallelization: run independent jobs concurrently.
- Caching: cache dependencies, Docker layers, build artifacts between runs.
- Branch protection: require CI pass before merge.

CONTINUOUS DELIVERY:
- Artifact promotion: same artifact through dev → staging → production.
- Environment parity: staging should mirror production as closely as possible.
- Automated testing gates: unit, integration, e2e, performance tests.
- Manual approval gates for production (when required).
- Rollback: automated rollback on health check failure.

CI/CD PLATFORMS:
- GitHub Actions: tight GitHub integration, marketplace actions.
- GitLab CI: built-in, powerful pipeline syntax, Auto DevOps.
- Jenkins: self-hosted, highly customizable, plugin ecosystem.
- CircleCI: fast, Docker-native, good caching.
- ArgoCD: GitOps continuous delivery for Kubernetes.

DEPLOYMENT STRATEGIES:
- Rolling update: gradually replace instances. Default for Kubernetes.
- Blue-green: run two identical environments, switch traffic instantly.
- Canary: route small percentage of traffic to new version, monitor, expand.
- Feature flags: deploy code behind flags, enable for specific users.
- A/B testing: route traffic to variants for comparison.
</ci_cd>

<infrastructure_as_code>
TOOLS:
- Terraform: multi-cloud, declarative, state management, module ecosystem.
- Pulumi: infrastructure in general-purpose languages (Python, TypeScript).
- CloudFormation: AWS-native, deep service integration.
- CDK: write CloudFormation in TypeScript/Python.
- Ansible: configuration management and orchestration.

BEST PRACTICES:
- Modular design: reusable modules for common infrastructure patterns.
- State management: remote state with locking (S3+DynamoDB, Terraform Cloud).
- Environments: separate state per environment, promote configurations.
- Plan before apply: always review the plan before making changes.
- Drift detection: regularly check for manual changes that diverge from code.
- Secret management: never store secrets in code. Use Vault, AWS Secrets Manager.
</infrastructure_as_code>

<observability>
THREE PILLARS:
- Metrics: numeric measurements over time (Prometheus, CloudWatch, Datadog).
- Logs: structured event records (ELK, Loki, CloudWatch Logs).
- Traces: request flow across services (Jaeger, Zipkin, Datadog APM).

MONITORING:
- Golden signals: latency, traffic, errors, saturation.
- SLIs/SLOs/SLAs: define and measure service level objectives.
- Alerting: alert on symptoms (SLO burn rate), not causes.
- Dashboards: operational dashboards with clear hierarchy.
- On-call: rotation, escalation, runbooks for common alerts.

INCIDENT MANAGEMENT:
- Detection: automated alerting catches issues before users report them.
- Response: clear roles (incident commander, communicator, responders).
- Mitigation: quick fixes to restore service (rollback, scale, redirect).
- Resolution: permanent fix after service is restored.
- Post-mortem: blameless analysis, action items, timeline documentation.
</observability>
