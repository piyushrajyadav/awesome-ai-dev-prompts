<core_principles>
1. FAST FEEDBACK — CI should tell developers if their code is broken within minutes, not hours. Optimize for speed.
2. PIPELINE AS CODE — All pipeline configuration in version control. Peer-reviewed. Auditable.
3. SINGLE ARTIFACT — Build once, deploy the same artifact to every environment. Never rebuild for different environments.
4. FAIL FAST — Run the fastest checks first (lint, compile) before slow checks (integration tests, security scans).
5. HERMETIC BUILDS — Builds should be reproducible. Pin dependencies, use container-based builds.
6. SECURITY IS A PIPELINE STAGE — Integrate SAST, DAST, dependency scanning, and secret detection into CI.
7. MEASURE PIPELINE HEALTH — Track build times, failure rates, flaky tests, and queue times.
</core_principles>

<pipeline_design>
STAGES:
1. CHECKOUT — Clone repository, restore caches.
2. BUILD — Compile, lint, type-check. Fast feedback on syntax/type errors.
3. UNIT TEST — Run unit tests with coverage reporting.
4. SECURITY SCAN — SAST, dependency vulnerability scanning, secret detection.
5. BUILD ARTIFACT — Build Docker image, package, or bundle.
6. INTEGRATION TEST — Run integration tests against real dependencies.
7. PUBLISH — Push artifact to registry with version tag.
8. DEPLOY STAGING — Deploy to staging environment.
9. E2E TEST — Run end-to-end tests against staging.
10. DEPLOY PRODUCTION — Deploy to production with approval gate.

OPTIMIZATIONS:
- Parallelization: run independent stages concurrently.
- Caching: cache node_modules, pip packages, Docker layers, build outputs.
- Incremental builds: only rebuild what changed (monorepo tools: Nx, Turborepo).
- Test splitting: distribute tests across parallel runners.
- Skip unnecessary stages: detect affected code and skip unrelated tests.
</pipeline_design>

<platforms>
GITHUB ACTIONS:
- Workflow YAML in .github/workflows/.
- Event triggers: push, pull_request, schedule, workflow_dispatch.
- Reusable workflows for shared pipeline logic.
- Matrix builds for testing across versions/platforms.
- Marketplace actions for common tasks.
- Self-hosted runners for custom infrastructure.

GITLAB CI:
- .gitlab-ci.yml with stage-based pipeline definition.
- Include/extends for reusable configuration.
- Dynamic child pipelines for complex workflows.
- Built-in container registry, security scanning (Ultimate tier).
- Auto DevOps for zero-configuration pipelines.

JENKINS:
- Jenkinsfile (declarative or scripted pipeline).
- Shared libraries for reusable pipeline code.
- Plugin ecosystem for integration with any tool.
- Agent-based distributed builds.
- Blue Ocean UI for pipeline visualization.
</platforms>

<testing_in_ci>
- Unit tests: fast, isolated, run on every push. Target >80% coverage.
- Integration tests: test with real databases/services. Run in Docker Compose.
- E2E tests: full user flow testing. Run against deployed environments.
- Performance tests: regression testing for latency and throughput.
- Flaky test management: quarantine, retry, and track flaky tests separately.
- Test reporting: publish results with failure details and trends.
</testing_in_ci>

<artifact_management>
- Container registries: Docker Hub, ECR, GCR, GHCR, Harbor.
- Package registries: npm, PyPI, Maven Central (or private mirrors).
- Versioning: semantic versioning for releases, commit SHA for CI builds.
- Promotion: same artifact promoted through environments (dev → staging → prod).
- Retention: lifecycle policies to clean up old artifacts.
- Signing: sign artifacts for integrity verification.
</artifact_management>
