<core_principles>
1. GARBAGE IN, GARBAGE OUT — The quality of your RAG system is bounded by the quality of your document processing and chunking. Invest heavily here.
2. RETRIEVAL QUALITY IS EVERYTHING — If you don't retrieve the right context, the LLM cannot generate the right answer. Optimize retrieval before generation.
3. CHUNK SIZE MATTERS — Chunks too small lose context, chunks too large dilute relevance. Optimize chunk size empirically for your domain.
4. HYBRID SEARCH WINS — Combine dense (semantic) and sparse (keyword/BM25) retrieval. Neither alone is sufficient.
5. RE-RANKING IS HIGH-ROI — A cross-encoder re-ranker on top-K results dramatically improves final retrieval quality.
6. EVALUATE END-TO-END — Evaluate both retrieval (recall@K, MRR) and generation (faithfulness, relevance) separately and together.
7. CITE YOUR SOURCES — Always provide source references so users can verify answers.
</core_principles>

<document_processing>
INGESTION:
- PDF: use unstructured, PyPDF2, or specialized parsers. Preserve layout for tables.
- HTML: parse with BeautifulSoup, preserve semantic structure.
- Office docs: python-docx, openpyxl for structured extraction.
- OCR: Tesseract or cloud vision APIs for scanned documents.
- Tables: extract tables into structured format, don't chunk them as text.
- Images: caption images with vision models for searchability.

CHUNKING STRATEGIES:
- Fixed-size chunks: simple but may split semantic units. Use with overlap.
- Semantic chunking: split at paragraph/section boundaries. Preserve context.
- Recursive character splitting: LangChain's default. Respects hierarchy.
- Document-structure-aware: use headings, sections, and document outline.
- Sentence-window: small chunks for retrieval, expand to larger window for context.
- Parent-child: retrieve child chunks, pass parent chunks to LLM.
- Chunk size: typically 256-1024 tokens. Test empirically.
- Overlap: 10-20% overlap between consecutive chunks prevents context loss.

METADATA:
- Attach source file, page number, section heading, date, author to each chunk.
- Enable metadata filtering in retrieval (e.g., filter by document type or date).
- Store hierarchical metadata for drill-down capabilities.
</document_processing>

<retrieval_architecture>
EMBEDDING:
- Choose embedding model matching your domain and languages.
- Models: OpenAI text-embedding-3, Cohere embed-v3, BGE, E5, GTE.
- Normalize embeddings for cosine similarity.
- Fine-tune embeddings on domain data for best results.

VECTOR DATABASES:
- Pinecone: fully managed, scalable, metadata filtering.
- Weaviate: hybrid search built-in, GraphQL API.
- Milvus/Zilliz: high-performance, open-source with cloud option.
- Qdrant: Rust-based, filtering, payload support.
- pgvector: PostgreSQL extension for simple deployments.
- ChromaDB: lightweight, good for prototyping.

RETRIEVAL STRATEGIES:
- Dense retrieval: semantic similarity with embeddings.
- Sparse retrieval: BM25 keyword matching (Elasticsearch, Lucene).
- Hybrid: combine dense + sparse scores (reciprocal rank fusion).
- Multi-query: rephrase user query into multiple variants, merge results.
- HyDE: generate hypothetical answer, use its embedding for retrieval.
- Query decomposition: break complex queries into sub-queries.

RE-RANKING:
- Cross-encoder re-ranker: MUCH more accurate than bi-encoder alone.
- Cohere Rerank, BGE-reranker, ms-marco-MiniLM cross-encoder.
- Re-rank top 20-50 results, pass top 3-5 to LLM.
- Significant accuracy improvement for marginal latency cost.
</retrieval_architecture>

<generation>
- System prompt: instruct LLM to answer ONLY from provided context.
- Context formatting: present retrieved chunks with source labels.
- Faithfulness: detect and prevent hallucination (answers not grounded in context).
- "I don't know": instruct model to say when context doesn't contain the answer.
- Citation: generate inline citations linking to source chunks.
- Conversational: maintain chat history, resolve coreferences in queries.
</generation>

<evaluation>
RETRIEVAL METRICS:
- Recall@K: fraction of relevant docs in top K results.
- MRR: mean reciprocal rank of the first relevant result.
- NDCG: normalized discounted cumulative gain.
- Target: Recall@10 > 0.95 for most applications.

GENERATION METRICS:
- Faithfulness: is the answer grounded in the retrieved context?
- Relevance: does the answer address the user's question?
- Completeness: are all relevant points from the context included?
- Use frameworks: RAGAS, DeepEval, TruLens for automated evaluation.

EVALUATION DATA:
- Build golden Q&A datasets with expected answers and source documents.
- Synthetic evaluation: use LLM to generate Q&A pairs from documents.
- Human evaluation for subjective quality assessment.
</evaluation>
