<core_principles>
1. LLMS ARE PROBABILISTIC — Every LLM call can fail or produce unexpected output. Design for unreliability with validation, retries, and fallbacks.
2. STRUCTURED OUTPUTS — Always use structured output formats (JSON, XML) with schema validation. Never parse free-text responses in production.
3. PROMPT ENGINEERING IS SOFTWARE ENGINEERING — Version prompts, test them systematically, measure their performance, and review changes.
4. COST-AWARE — Track token usage, optimize prompt length, use model routing (cheap model first, expensive model as fallback).
5. LATENCY MATTERS — LLM calls are slow. Stream responses, cache results, and run independent calls in parallel.
6. EVALUATION IS ESSENTIAL — Build automated evaluation pipelines. Test prompts against golden datasets before deploying changes.
7. SECURITY — Validate inputs to prevent prompt injection. Never expose raw model outputs without post-processing.
</core_principles>

<prompt_engineering>
FUNDAMENTALS:
- System prompts: define persona, rules, and output format.
- Few-shot examples: provide 3-5 examples for consistent formatting.
- Chain-of-thought: ask the model to reason step-by-step for complex tasks.
- Role prompting: assign specific expert roles for domain tasks.

STRUCTURED OUTPUT:
- Define JSON schemas for all outputs.
- Use function calling / tool use for reliable structured extraction.
- Implement output validation with Pydantic / Zod schemas.
- Retry with error feedback when output doesn't match schema.

ADVANCED TECHNIQUES:
- Prompt chaining: break complex tasks into sequential simple prompts.
- Self-consistency: sample multiple responses and pick the majority answer.
- Retrieval-augmented generation: inject relevant context from a knowledge base.
- Decomposition: split complex problems into sub-problems solved independently.
</prompt_engineering>

<architecture_patterns>
LLM PIPELINE:
- Input validation → preprocessing → prompt construction → API call → output parsing → validation → post-processing.
- Each stage should be independently testable and monitorable.

ORCHESTRATION:
- Model routing: use small/cheap models for simple tasks, large models for complex ones.
- Fallback chains: primary model → fallback model → cached response → graceful degradation.
- Parallel execution: run independent LLM calls concurrently.
- Streaming: stream tokens to users for perceived speed improvement.

RELIABILITY:
- Retry with exponential backoff for API failures.
- Circuit breakers for persistent model failures.
- Output validation with automatic retry and prompt correction.
- Rate limiting and token budget management.
- Timeout management with graceful cancellation.

CACHING:
- Semantic caching: cache similar (not just identical) queries.
- Response caching with TTL based on data freshness requirements.
- Embedding-based cache lookup for semantic similarity.
</architecture_patterns>

<evaluation_and_testing>
- Golden datasets: curated input/output pairs for regression testing.
- A/B testing: compare prompt versions on real traffic.
- LLM-as-judge: use a strong model to evaluate outputs of another model.
- Automated metrics: format compliance rate, factual accuracy, relevance scores.
- Latency and cost tracking per prompt version.
- Red-teaming: adversarial testing for prompt injection and edge cases.
</evaluation_and_testing>
