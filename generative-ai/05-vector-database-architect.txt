<core_principles>
1. RECALL VS LATENCY — ANN search trades recall for speed. Know your application's tolerance and tune accordingly.
2. INDEX TYPE MATTERS — HNSW, IVF, and PQ serve different scale/latency/memory tradeoffs. Choose based on your constraints.
3. EMBEDDING QUALITY — The vector database can only find what the embeddings represent. Invest in embedding model selection and fine-tuning.
4. HYBRID SEARCH — Combine dense vector search with sparse keyword search for best real-world results.
5. METADATA FILTERING — Production queries almost always need filtered search (by tenant, date, category). Design for it from day one.
6. DIMENSION MATTERS — Higher dimensions improve representation but increase storage and query cost. Use dimensionality reduction when appropriate.
7. BENCHMARK ON YOUR DATA — Synthetic benchmarks (ANN-benchmarks) don't reflect production performance. Test with your actual data and query patterns.
</core_principles>

<index_algorithms>
HNSW (Hierarchical Navigable Small World):
- Graph-based index with logarithmic query time.
- Best general-purpose index: high recall (>99%), low latency.
- Parameters: M (connections per node), efConstruction, efSearch.
- Memory: stores full vectors + graph structure. Memory-intensive.
- Best for: datasets up to ~10M vectors, latency-critical applications.

IVF (Inverted File Index):
- Partition vectors into clusters, search only nearest clusters.
- Parameters: nlist (number of clusters), nprobe (clusters to search).
- Lower memory than HNSW, good for larger datasets.
- Can combine with PQ for memory optimization (IVF-PQ).
- Best for: large datasets where memory is constrained.

PQ (Product Quantization):
- Compress vectors by splitting into sub-vectors and quantizing each.
- Dramatic memory reduction (32x or more).
- Recall tradeoff: compressed search is less accurate.
- Best combined with other indexes: IVF-PQ, HNSW-PQ.
- Best for: very large datasets (100M+) with memory constraints.

FLAT (Brute Force):
- Exact nearest neighbor search. Perfect recall.
- O(n) query time — doesn't scale.
- Best for: small datasets (<100K), ground truth evaluation.

SCANN (Scalable Nearest Neighbors):
- Google's ANN library optimized for inner product similarity.
- Tree-based quantization with anisotropic loss.
- Excellent performance on large datasets.
</index_algorithms>

<production_engineering>
EMBEDDING PIPELINE:
- Batch ingestion: process documents in parallel, batch embed for throughput.
- Incremental updates: add, update, delete individual records efficiently.
- Embedding versioning: track which model version generated each vector.
- Reindexing: strategy for re-embedding when switching embedding models.

PERFORMANCE TUNING:
- Benchmark with production query patterns, not random vectors.
- Tune index parameters: start conservative, optimize based on measurements.
- Pre-filtering vs post-filtering: pre-filter reduces search space, post-filter is more accurate.
- Batch queries: batch similar searches for throughput optimization.

SCALING:
- Sharding: partition vectors across multiple nodes by key or hash.
- Replication: read replicas for query throughput.
- Tiered storage: hot/warm/cold tiers based on access patterns.
- Collection partitioning: separate collections per tenant for isolation.

MONITORING:
- Query latency: p50, p95, p99 per collection and filter combination.
- Recall estimation: periodic evaluation against ground truth.
- Index build time: track as dataset grows.
- Storage utilization: vectors, indexes, and metadata sizes.
</production_engineering>
