<core_principles>
1. EVENTS ARE IMMUTABLE FACTS — Model data as immutable event streams. Events represent things that happened and cannot be changed.
2. ORDERING IS HARD — Events arrive out of order. Design for event-time processing with watermarks, not processing-time.
3. EXACTLY-ONCE IS EXPENSIVE — Understand the delivery guarantees: at-most-once, at-least-once, exactly-once. Choose based on your requirements.
4. STATE MANAGEMENT IS THE CHALLENGE — Stateful stream processing (joins, aggregations, windows) requires careful state management and fault tolerance.
5. BACKPRESSURE — Systems must handle throughput spikes. Implement backpressure to prevent cascading failures.
6. SCHEMA EVOLUTION — Event schemas change over time. Use schema registry and backward/forward compatible formats.
7. REPLAYABILITY — Design systems that can replay events from any point for debugging, reprocessing, and recovery.
</core_principles>

<event_streaming>
APACHE KAFKA:
- Distributed event log: append-only, partitioned, replicated.
- Topics: logical event channels. Partitions: parallelism unit.
- Consumer groups: parallel consumption with automatic partition assignment.
- Retention: time-based or size-based. Compacted topics for key-value state.
- Schema Registry: Avro/Protobuf/JSON Schema for event validation.
- Connect: pre-built connectors for sources and sinks.

KAFKA BEST PRACTICES:
- Partition key: choose carefully — determines ordering and parallelism.
- Number of partitions: plan for peak throughput (each partition ~10MB/s).
- Replication factor: 3 for production. min.insync.replicas = 2.
- Idempotent producers: enable.idempotence = true for exactly-once writes.
- Consumer lag monitoring: critical operational metric.

OTHER PLATFORMS:
- Amazon Kinesis: AWS-managed streaming. Simpler but less flexible than Kafka.
- Google Pub/Sub: GCP-managed messaging with automatic scaling.
- Azure Event Hubs: Azure-managed, Kafka-compatible API.
- Apache Pulsar: multi-tenancy, tiered storage, geo-replication.
- Redpanda: Kafka-compatible, C++ implementation, simpler operations.
</event_streaming>

<stream_processing>
APACHE FLINK:
- True stream processing: processes events one at a time (not micro-batch).
- Event-time processing with watermarks for handling late data.
- Exactly-once state management with checkpointing.
- SQL / Table API for declarative stream processing.
- Best for: complex event processing, stateful aggregations, low-latency.

SPARK STRUCTURED STREAMING:
- Micro-batch model: processes small batches at short intervals.
- Unified batch and streaming API in Spark.
- Good for: analytics, ML feature computation, moderate latency (seconds).
- Watermark support for late data handling.

KAFKA STREAMS:
- Library (not framework): runs as part of your application.
- No separate cluster needed — leverages Kafka for coordination.
- KTable / KStream abstractions for streams and tables.
- Good for: microservice-embedded stream processing.

PROCESSING PATTERNS:
- Filtering: select events matching criteria.
- Mapping: transform event structure.
- Windowing: tumbling, sliding, session windows for bounded aggregations.
- Joins: stream-stream, stream-table joins with temporal semantics.
- Aggregation: counts, sums, averages over windows.
- CEP (Complex Event Processing): detect patterns across events.
</stream_processing>

<windowing_and_time>
WINDOW TYPES:
- Tumbling: fixed-size, non-overlapping windows (e.g., every 5 minutes).
- Sliding: fixed-size, overlapping windows (e.g., 5 min window, 1 min slide).
- Session: gap-based windows. Opens on first event, closes after inactivity.
- Global: single window for all time (use with triggers).

TIME SEMANTICS:
- Event time: when the event actually happened (from event payload).
- Processing time: when the system processes the event (wall clock).
- Ingestion time: when the event enters the streaming system.
- Always prefer event time for correct results.

WATERMARKS:
- Signal of event-time progress: "no more events before time T expected."
- Allow systems to trigger window computations.
- Late data: events arriving after watermark. Handle with allowed lateness.
- Trade-off: aggressive watermarks miss late data, conservative ones add latency.
</windowing_and_time>
