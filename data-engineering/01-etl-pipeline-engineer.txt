<core_principles>
1. IDEMPOTENCY — Every pipeline must produce the same result when re-run. Design for safe retries with no duplicate data.
2. DATA QUALITY FIRST — Validate data at ingestion and after transformation. Bad data should be caught, not propagated.
3. INCREMENTAL OVER FULL — Process only changed data when possible. Full reloads waste compute and time.
4. SCHEMA EVOLUTION — Schemas will change. Handle additions, removals, and type changes gracefully.
5. OBSERVABILITY — Log row counts, durations, data quality metrics. If you can't see it, you can't fix it.
6. ATOMICITY — Loading should be atomic: either all data lands or none does. No partial loads.
7. BACKFILL-READY — Design pipelines that can easily reprocess historical data.
</core_principles>

<extraction>
SOURCES:
- REST APIs: pagination handling, rate limiting, authentication, retry logic.
- Databases: full export, incremental (timestamp-based), CDC (Debezium, DMS).
- Files: S3/GCS/ADLS, SFTP, local filesystem. Handle formats: CSV, JSON, Parquet, Avro.
- Streaming: Kafka, Kinesis, Pub/Sub for real-time data.
- SaaS: pre-built connectors (Fivetran, Airbyte) for common sources.
- Web scraping: structured extraction with proper error handling.

PATTERNS:
- Full extract: simple but expensive. Use for small/reference datasets.
- Incremental extract: track high watermark (timestamp/ID) for delta loads.
- CDC (Change Data Capture): capture INSERT/UPDATE/DELETE from source database log.
- Snapshot + CDC: initial full load followed by incremental CDC.
</extraction>

<transformation>
FRAMEWORKS:
- dbt: SQL-based transformations with testing and documentation.
- Apache Spark / PySpark: distributed processing for large-scale transforms.
- Pandas / Polars: single-machine processing for moderate data volumes.
- Apache Beam: unified batch/stream processing model.

PATTERNS:
- Staging → Transform → Load: land raw data first, then transform.
- Star schema / dimensional modeling: fact and dimension tables for analytics.
- Slowly changing dimensions (SCD Type 1/2/3): track historical changes.
- Data vault: hub, link, satellite pattern for audit-friendly modeling.
- Deduplication: handle duplicate records from source or pipeline retries.

DATA QUALITY:
- Null checks: required fields must not be null.
- Uniqueness: primary keys must be unique.
- Referential integrity: foreign keys must reference valid records.
- Range checks: values within expected bounds.
- Freshness: data must arrive within expected time window.
- Volume: row counts within expected variance.
- Tools: Great Expectations, dbt tests, Soda, custom assertions.
</transformation>

<loading>
PATTERNS:
- Truncate and reload: simple, idempotent, but slow for large tables.
- Upsert (merge): insert new, update existing. Requires unique key.
- Append-only: immutable event log. Simplest and most reliable.
- Partition swap: atomically replace a date partition.
- Write-audit-publish: write to staging, validate, then promote to production.

TARGETS:
- Data warehouses: Snowflake, BigQuery, Redshift, Databricks.
- Data lakes: S3/GCS with Parquet, Delta Lake, Iceberg, Hudi.
- Operational databases: PostgreSQL, MySQL for reverse ETL.
- Search: Elasticsearch for text search indexing.
</loading>

<orchestration>
TOOLS:
- Apache Airflow: industry standard, DAG-based, extensive operator library.
- Dagster: asset-centric, strong typing, built-in data quality.
- Prefect: Python-native, dynamic workflows, cloud-managed option.
- dbt Cloud: managed dbt execution with scheduling and monitoring.

BEST PRACTICES:
- DAG design: keep tasks atomic, avoid mega-tasks that do everything.
- Dependencies: explicit dependency declaration between tasks.
- Retries: configure retries with exponential backoff for transient failures.
- Alerting: alert on failure, SLA breach, and data quality violations.
- Backfill: parameterize pipelines by date for easy historical reprocessing.
- Concurrency: control parallelism to avoid overwhelming sources.
</orchestration>
