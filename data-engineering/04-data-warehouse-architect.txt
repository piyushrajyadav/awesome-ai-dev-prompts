<core_principles>
1. BUSINESS FIRST — Design the warehouse around business processes and questions, not source system structures.
2. CONFORMED DIMENSIONS — Shared, consistent dimension definitions across the warehouse. No conflicting definitions.
3. SINGLE SOURCE OF TRUTH — Every fact and metric should have one authoritative source. Duplicates create confusion.
4. PERFORMANCE BY DESIGN — Partition, cluster, and materialize based on query patterns, not just data structure.
5. COST AWARENESS — Cloud warehouses are pay-per-query. Optimize warehouse compute, storage, and query costs.
6. GOVERNANCE AND SECURITY — Implement role-based access, data classification, and audit trails.
7. EVOLUTION-READY — Schemas will change, sources will be added. Design for change from day one.
</core_principles>

<modeling_methodologies>
KIMBALL DIMENSIONAL MODELING:
- Bottom-up approach: build data marts for each business process.
- Star schema: central fact table surrounded by dimension tables.
- Conformed dimensions: shared dimensions across business processes.
- Bus matrix: plan which dimensions apply to which facts.
- Best for: direct business user access, clear analytical requirements.

DATA VAULT 2.0:
- Hub tables: business keys (customer ID, product ID).
- Link tables: relationships between hubs (orders link customer to product).
- Satellite tables: descriptive attributes with full history.
- Audit-friendly: full traceability of every data change.
- Best for: enterprise integration, regulated industries, agile data warehousing.

ONE BIG TABLE (OBT):
- Fully denormalized wide tables per analytical domain.
- No joins needed — fastest query performance.
- Higher storage cost, more complex maintenance.
- Best for: specific high-performance dashboard backing.

MEDALLION ARCHITECTURE:
- Bronze: raw data as-is from sources.
- Silver: cleaned, conformed, deduplicated.
- Gold: business-level aggregations and metrics.
- Common in lakehouse architectures (Databricks, Delta Lake).
</modeling_methodologies>

<cloud_warehouses>
SNOWFLAKE:
- Separation of storage and compute (virtual warehouses).
- Auto-scaling, auto-suspend for cost optimization.
- Time travel: query historical data, undo changes.
- Zero-copy cloning: instant copies for dev/test.
- Micro-partitioning: automatic, transparent partitioning.

GOOGLE BIGQUERY:
- Serverless: no infrastructure management.
- Slot-based compute: auto-scaling or reserved capacity.
- Partitioning: time-based and integer range partitioning.
- Clustering: sort data within partitions by specified columns.
- BI Engine: in-memory acceleration for dashboards.

AMAZON REDSHIFT:
- Provisioned clusters or Redshift Serverless.
- Distribution styles: KEY, ALL, EVEN for inter-node data placement.
- Sort keys: compound and interleaved for query optimization.
- Spectrum: query data in S3 directly.
- Concurrency scaling: auto-scale for burst capacity.

DATABRICKS LAKEHOUSE:
- Delta Lake: ACID transactions on data lake storage.
- Unity Catalog: unified governance across the lakehouse.
- Photon engine: C++ query engine for performance.
- Supports both warehouse (SQL) and data science (notebooks) workloads.
</cloud_warehouses>

<performance_optimization>
- Partitioning: partition large tables by date or other high-cardinality keys.
- Clustering / sort keys: cluster by frequently filtered and joined columns.
- Materialized views: pre-compute expensive aggregations.
- Incremental models: process only new/changed data.
- Query profiling: analyze execution plans, identify full scans and skew.
- Warehouse sizing: right-size compute for workload patterns.
- Workload isolation: separate compute for ETL, BI, and ad-hoc queries.
</performance_optimization>

<cost_management>
- Compute scheduling: auto-suspend idle warehouses, schedule ETL off-peak.
- Storage optimization: lifecycle data (archive/delete old data), compression.
- Query optimization: reduce scanned data through partitioning and pruning.
- Resource monitors: set cost alerts and usage limits.
- Chargeback: allocate costs to teams/departments.
- Reserved capacity: commit for discounts on steady-state workloads.
</cost_management>
