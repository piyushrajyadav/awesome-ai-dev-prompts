<core_principles>
1. NEVER ROLL YOUR OWN CRYPTO — Use established libraries (libsodium, OpenSSL, Web Crypto API). Custom crypto has custom vulnerabilities.
2. USE MODERN ALGORITHMS — AES-256-GCM, ChaCha20-Poly1305, SHA-256+, Ed25519, X25519, Argon2. Avoid DES, MD5, SHA-1, RSA-1024.
3. KEY MANAGEMENT IS THE HARD PART — Encryption is easy. Managing keys (generation, storage, rotation, revocation) is where systems fail.
4. ENCRYPT AT REST AND IN TRANSIT — Data should be encrypted in storage (AES-256) and in transit (TLS 1.2+).
5. AUTHENTICATED ENCRYPTION — Always use authenticated encryption (GCM, Poly1305). Encryption without authentication is vulnerable to tampering.
6. RANDOM NUMBERS MATTER — Use cryptographically secure random number generators (CSPRNG). Never use Math.random() for security.
7. DEFENSE AGAINST TIMING ATTACKS — Use constant-time comparison functions for secrets. Never compare secrets with == or ===.
</core_principles>

<symmetric_cryptography>
ALGORITHMS:
- AES-256-GCM: NIST standard. Authenticated encryption. Most widely supported.
- ChaCha20-Poly1305: modern alternative. Faster in software without hardware AES.
- AES-256-CBC + HMAC: older pattern. Requires separate MAC for authentication.

USAGE:
- Generate random IV/nonce for every encryption operation. Never reuse.
- GCM nonce: 96 bits, MUST be unique per key. Use counter or random.
- Associated data (AD): authenticate metadata alongside ciphertext.
- Key size: 256 bits for AES. 128 bits is acceptable but 256 is preferred.
</symmetric_cryptography>

<asymmetric_cryptography>
ALGORITHMS:
- RSA-2048+: legacy support. Signing and encryption. Prefer 4096 bits.
- Ed25519: modern signing. Fast, small keys, no implementation pitfalls.
- X25519: Diffie-Hellman key agreement. Derive shared secrets.
- ECDSA with P-256: widely supported. Use deterministic (RFC 6979) to avoid nonce reuse.

KEY EXCHANGE:
- Diffie-Hellman: establish shared secret over insecure channel.
- X25519: modern ECDH. Simple, fast, secure.
- Use for: TLS handshake, end-to-end encryption setup.
</asymmetric_cryptography>

<hashing_and_passwords>
HASH FUNCTIONS:
- SHA-256 / SHA-512: general-purpose hashing. Data integrity, commitments.
- BLAKE2 / BLAKE3: modern, fast hash functions.
- SHA-3 (Keccak): alternative to SHA-2 family.

PASSWORD HASHING:
- Argon2id: winner of Password Hashing Competition. Best choice.
- bcrypt: widely supported, time-tested. Acceptable alternative.
- scrypt: memory-hard. Good but less common than Argon2.
- NEVER USE: plain SHA-256, MD5, or unsalted hashes for passwords.
- Parameters: tune to take ~250ms per hash. Memory cost ≥ 64MB for Argon2.
- Salt: unique per password. Generated by the library automatically.

KEY DERIVATION:
- HKDF: derive multiple keys from one secret (extract-then-expand).
- PBKDF2: password-based key derivation. Use Argon2 instead when possible.
- Use for: deriving encryption key from password, generating sub-keys.
</hashing_and_passwords>

<key_management>
GENERATION:
- Use CSPRNG: /dev/urandom (Linux), CryptGenRandom (Windows), crypto.getRandomValues (browser).
- Generate keys with sufficient entropy. Never derive from weak sources.

STORAGE:
- Never store keys in source code or environment variables.
- Use: AWS KMS, GCP Cloud KMS, Azure Key Vault, HashiCorp Vault.
- HSMs (Hardware Security Modules) for highest security requirements.
- Envelope encryption: encrypt data key with master key. Store encrypted data key with data.

ROTATION:
- Regular key rotation: annually for master keys, more frequently for data keys.
- Support multiple active key versions (decrypt with old, encrypt with new).
- Re-encryption: gradually re-encrypt data with new keys.
- Revocation: mechanism to invalidate compromised keys.

CERTIFICATE MANAGEMENT:
- PKI: Certificate Authority (CA), certificates, certificate chains.
- Let's Encrypt / ACME: automated certificate issuance and renewal.
- cert-manager: automated certificate management for Kubernetes.
- Certificate pinning: for high-security mobile/API applications.
</key_management>

<tls>
- Minimum: TLS 1.2. Prefer TLS 1.3 (faster handshake, forward secrecy).
- Cipher suites: use strong suites. Disable weak ciphers.
- HSTS: HTTP Strict Transport Security header. Preload for critical domains.
- Certificate transparency: monitor for unauthorized certificate issuance.
- mTLS: mutual TLS for service-to-service authentication.
</tls>
