<core_principles>
1. DEFENSE IN DEPTH — Multiple layers of security controls. No single control is sufficient.
2. LEAST PRIVILEGE — Grant minimum required access. Default deny, explicit allow.
3. FAIL SECURELY — When systems fail, they should fail closed (deny access), not open.
4. INPUT VALIDATION — Never trust user input. Validate, sanitize, and encode at every boundary.
5. SHIFT LEFT — Integrate security early: threat modeling in design, SAST in CI, security code review in PRs.
6. SECURE DEFAULTS — Systems should be secure out of the box. Security shouldn't require configuration.
7. ASSUME BREACH — Design systems assuming attackers will get in. Limit blast radius.
</core_principles>

<owasp_top_10>
A01 BROKEN ACCESS CONTROL:
- Enforce authorization on every endpoint, not just the UI.
- Implement RBAC or ABAC consistently.
- Check ownership: users should only access their own resources (prevent IDOR).
- Deny by default. Require explicit permission grants.

A02 CRYPTOGRAPHIC FAILURES:
- Encrypt data at rest and in transit (TLS 1.2+).
- Use strong algorithms: AES-256, SHA-256+, RSA-2048+, Ed25519.
- Never roll your own crypto. Use established libraries.
- Manage keys properly: rotation, access control, HSMs for critical keys.

A03 INJECTION:
- Parameterized queries for SQL (never string concatenation).
- ORM with parameterized queries for database access.
- Context-aware output encoding (HTML, JS, URL, CSS).
- Command injection: avoid shell execution, use library functions.

A04 INSECURE DESIGN:
- Threat modeling during design phase (STRIDE, DREAD).
- Abuse case analysis: how could this feature be misused?
- Rate limiting on sensitive operations.
- Business logic validation on the server side.

A05 SECURITY MISCONFIGURATION:
- Remove default credentials and unnecessary features.
- Security headers: CSP, X-Frame-Options, HSTS, X-Content-Type-Options.
- Disable directory listing, debug modes, stack traces in production.
- Automated configuration scanning and hardening.

A06 VULNERABLE COMPONENTS:
- Dependency scanning in CI (Dependabot, Snyk, Renovate).
- Keep dependencies updated. Automated update PRs.
- SBOM (Software Bill of Materials) for component tracking.
- Pin dependency versions for reproducible builds.

A07 AUTH FAILURES:
- Strong password policies with breach detection (HaveIBeenPwned).
- Multi-factor authentication for sensitive operations.
- Secure session management: HttpOnly, Secure, SameSite cookies.
- Account lockout and brute force protection.

A08 DATA INTEGRITY FAILURES:
- Verify integrity of updates and data from untrusted sources.
- Code signing for software distribution.
- CI/CD pipeline security: protect build artifacts and deployment credentials.

A09 LOGGING AND MONITORING:
- Log security events: authentication, authorization, data access.
- Don't log sensitive data (passwords, tokens, PII).
- Alerting on suspicious patterns: brute force, privilege escalation.
- Incident detection and response capabilities.

A10 SSRF (Server-Side Request Forgery):
- Validate and restrict outbound URLs from user input.
- Block requests to internal networks and metadata endpoints.
- Use allowlists for permitted external domains.
</owasp_top_10>

<threat_modeling>
STRIDE:
- Spoofing: can someone impersonate a user or system?
- Tampering: can data be modified in transit or at rest?
- Repudiation: can actions be performed without accountability?
- Information Disclosure: can sensitive data be exposed?
- Denial of Service: can the system be overwhelmed?
- Elevation of Privilege: can users gain unauthorized access?

PROCESS:
1. Map the system: components, data flows, trust boundaries.
2. Identify threats using STRIDE for each component/flow.
3. Rate threats by risk (likelihood × impact).
4. Define mitigations for high-priority threats.
5. Validate mitigations are implemented correctly.
</threat_modeling>
