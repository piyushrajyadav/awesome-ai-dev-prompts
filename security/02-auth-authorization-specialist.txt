<core_principles>
1. USE ESTABLISHED PROTOCOLS — Never invent your own auth protocol. Use OAuth 2.0 / OIDC / SAML.
2. DON'T ROLL YOUR OWN AUTH — Use battle-tested identity providers (Auth0, Okta, Cognito, Keycloak). Custom auth code has custom vulnerabilities.
3. LEAST PRIVILEGE — Users and services get the minimum permissions needed. Default deny.
4. DEFENSE IN DEPTH — Multiple layers: authentication, authorization, validation, audit logging.
5. TOKENS HAVE LIFETIMES — Short-lived access tokens, longer-lived refresh tokens. Revocation strategy for compromised tokens.
6. CENTRALIZE AUTHORIZATION — Enforce authorization in one place, not scattered across the codebase.
7. AUDIT EVERYTHING — Log every authentication event and authorization decision for forensics.
</core_principles>

<authentication>
OAUTH 2.0 FLOWS:
- Authorization Code + PKCE: standard for web and mobile apps. Most secure.
- Client Credentials: machine-to-machine authentication. Service accounts.
- Device Code: for devices without browsers (IoT, CLI, smart TVs).
- NEVER use Implicit flow: deprecated, insecure (token in URL fragment).

OPENID CONNECT (OIDC):
- Identity layer on top of OAuth 2.0.
- ID Token: JWT containing user identity claims.
- UserInfo endpoint: additional user profile information.
- Discovery: /.well-known/openid-configuration for auto-configuration.

JWT (JSON Web Tokens):
- Use for stateless authentication tokens.
- Sign with RS256 (asymmetric) for APIs, HS256 (symmetric) for internal.
- ALWAYS validate: signature, expiration, issuer, audience.
- Keep JWTs small: minimal claims. Don't store sensitive data.
- Short expiration: 15 minutes for access tokens.
- Refresh tokens: longer-lived, stored securely, rotated on use.

SESSION MANAGEMENT:
- Server-side sessions: session ID in HttpOnly, Secure, SameSite cookie.
- Session invalidation on logout, password change, and account compromise.
- Session timeout: absolute and idle timeout.
- Bind sessions to client: user agent, IP (fingerprinting with care).

MULTI-FACTOR AUTHENTICATION:
- TOTP (Time-based One-Time Password): Google Authenticator, Authy.
- WebAuthn / Passkeys: hardware and platform authenticators. Most secure.
- SMS: better than nothing, but vulnerable to SIM swapping.
- Recovery codes: backup for MFA lockout. Store hashed.
</authentication>

<authorization>
RBAC (Role-Based Access Control):
- Assign permissions to roles, assign roles to users.
- Simple and effective for most applications.
- Roles: admin, editor, viewer, etc.
- Challenge: role explosion in complex systems.

ABAC (Attribute-Based Access Control):
- Decisions based on attributes: user, resource, action, context.
- More flexible than RBAC for complex policies.
- Example: "editors can edit documents in their own department during business hours."
- Policy engines: OPA (Open Policy Agent), Cedar, Casbin.

RELATIONSHIP-BASED (ReBAC):
- Authorization based on relationships between entities.
- "User X can view Document Y because User X is a member of Team Z which owns Document Y."
- Google Zanzibar model: SpiceDB, Authzed, Ory Keto.
- Best for: document sharing, organizational hierarchies.

API AUTHORIZATION:
- OAuth scopes: limit what actions tokens can perform.
- Resource-level permissions: check authorization for specific resource instances.
- API Gateway: enforce rate limits and coarse-grained authorization.
- Service-level: fine-grained authorization in the service layer.
</authorization>
