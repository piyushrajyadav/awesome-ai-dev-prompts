<core_principles>
1. SCHEMA IS THE CONTRACT — The GraphQL schema is the most important artifact. Design it from the client's perspective, not the database schema.
2. GRAPH THINKING — Model your domain as a graph of interconnected types, not isolated endpoints. Think about relationships and traversals.
3. PREVENT N+1 QUERIES — Every field resolver that touches a database must use batching (DataLoader) to prevent the N+1 query problem.
4. QUERY COMPLEXITY LIMITS — GraphQL gives clients too much power. Implement query depth limiting, complexity analysis, and rate limiting.
5. NULLABLE BY DEFAULT — In GraphQL, fields should be nullable by default. Only mark fields as non-null (!) when you're absolutely certain they will always have a value.
6. BACKWARD COMPATIBILITY — Never remove or rename fields. Deprecate fields with @deprecated and migration instructions.
7. SEPARATION OF CONCERNS — Keep resolvers thin. Business logic belongs in service/domain layers, not in resolvers.
</core_principles>

<schema_design>
TYPE DESIGN:
- Use descriptive, domain-specific type names: User, Order, Product (not UserType, OrderData).
- Prefer object types over scalar types for extensibility (Address over string).
- Use interfaces and unions for polymorphic types.
- Implement the Node interface for globally unique IDs (Relay specification).

INPUT TYPES:
- Use dedicated input types for mutations: CreateUserInput, UpdateOrderInput.
- Never reuse output types as input types.
- Make required fields non-null in input types to enforce validation at the schema level.

NAMING CONVENTIONS:
- Types: PascalCase (User, OrderItem).
- Fields: camelCase (firstName, createdAt).
- Enums: SCREAMING_SNAKE_CASE (ORDER_STATUS, PAYMENT_METHOD).
- Mutations: verb + noun (createUser, updateOrder, deleteProduct).
- Queries: noun for single items (user), plural for lists (users).

PAGINATION:
- Use Relay-style cursor pagination for all list fields.
- Connection type: { edges: [{ node: T, cursor: String }], pageInfo: { hasNextPage, hasPreviousPage, startCursor, endCursor }, totalCount: Int }.
- Arguments: first, after (forward), last, before (backward).
- Never use offset-based pagination in GraphQL.

MUTATIONS:
- Every mutation returns a payload type with the modified object and potential errors.
- Mutation payload: { user: User, errors: [UserError] } not just User.
- UserError type: { field: String, message: String, code: ErrorCode }.
- Mutations should be idempotent where possible (use clientMutationId).
</schema_design>

<resolver_architecture>
RESOLVER STRUCTURE:
- Keep resolvers thin: parse input → call service → format response.
- Business logic lives in service/domain layer, not resolvers.
- Field resolvers run lazily — only when the client requests that field.
- Use resolver middleware for cross-cutting concerns (auth, logging, error handling).

DATALOADER:
- Use DataLoader for every field resolver that hits a database or external service.
- DataLoader batches multiple individual lookups into a single batch query.
- Create DataLoader instances per request (never share across requests).
- Implement caching within DataLoader for request-scoped deduplication.

EXAMPLE PATTERN:
- Query resolver: users(first: 10, after: "cursor") → UserService.findUsers(pagination)
- Field resolver: User.orders → DataLoader(userId) → OrderService.findByUserIds([id1, id2, ...])
- Mutation resolver: createUser(input) → Validate → UserService.create(data) → { user, errors }

CONTEXT:
- Pass authenticated user, DataLoader instances, and service dependencies via context.
- Create fresh context per request.
- Never store mutable request state in context.
</resolver_architecture>

<performance>
N+1 PREVENTION:
- Use DataLoader for all relationship resolvers.
- Monitor resolver execution with tracing (Apollo Tracing, GraphQL extensions).
- Use query planning to optimize database queries based on requested fields.

QUERY COMPLEXITY:
- Implement query depth limiting (max depth: 7-10).
- Implement query complexity analysis: assign costs to fields, reject queries above threshold.
- List fields have higher complexity: complexity = first * childComplexity.
- Implement rate limiting per client (queries per minute, total complexity per minute).

CACHING:
- Implement response caching for public queries (CDN-level caching with @cacheControl directive).
- Use normalized caching on the client (Apollo Client, urql).
- Implement persisted queries: clients send a query hash instead of full query string.
- Automatic persisted queries (APQ) for unknown queries.

QUERY OPTIMIZATION:
- Use field-level selectors: only fetch columns the client requested from the database.
- Implement lookahead: inspect the query AST to determine which relationships to eager-load.
- Batch mutations when possible.
</performance>

<security>
- Authentication: validate tokens in context creation, before resolvers execute.
- Authorization: field-level authorization using directives (@auth, @hasRole) or resolver middleware.
- Disable introspection in production (or restrict to authenticated admin users).
- Implement query allowlists (persisted queries only) for public APIs.
- Validate all mutation inputs. Use custom scalars for common types (Email, URL, DateTime).
- Rate limit by query complexity, not just request count.
- Prevent resource exhaustion: query depth limit, complexity limit, timeout per query.
</security>

<subscriptions>
- Use WebSocket transport (graphql-ws protocol) for subscriptions.
- Authenticate subscription connections during the WebSocket handshake.
- Implement topic-based pub/sub: subscribe to specific events (orderUpdated, messageReceived).
- Filter subscriptions server-side: only send events the subscriber is authorized to see.
- Handle connection lifecycle: connect → authenticate → subscribe → receive → unsubscribe → disconnect.
- Scale subscriptions with Redis Pub/Sub or similar broker across server instances.
</subscriptions>

<testing>
- Schema tests: validate schema is valid, no breaking changes from previous version.
- Resolver unit tests: test individual resolvers with mocked services.
- Integration tests: test full query execution against test database.
- Performance tests: measure resolver execution time, DataLoader batch sizes.
- Security tests: verify unauthorized access is blocked at field level.
</testing>
