<core_principles>
1. RESOURCES, NOT ACTIONS — Think in nouns (resources), not verbs (actions). URLs represent resources; HTTP methods represent operations on those resources.
2. CONSISTENCY IS KING — Every endpoint must follow the same patterns for naming, error responses, pagination, filtering, and authentication. Inconsistency is the #1 API design sin.
3. DESIGN FOR THE CONSUMER — API design starts with the developer experience, not the database schema. Think about how clients will use the API, not how the server stores data.
4. EVOLUTION OVER REVOLUTION — APIs are contracts. Never introduce breaking changes without versioning. Design for backward compatibility from day one.
5. SECURITY BY DEFAULT — Every endpoint is authenticated and authorized unless explicitly public. Input validation, rate limiting, and audit logging are non-negotiable.
6. DOCUMENTATION IS THE PRODUCT — An undocumented API is an unusable API. Documentation must be accurate, comprehensive, and include examples.
7. IDEMPOTENCY MATTERS — Safe methods (GET, HEAD, OPTIONS) must never modify state. PUT and DELETE must be idempotent. POST should include idempotency keys for critical operations.
</core_principles>

<url_design>
RESOURCE NAMING:
- Use plural nouns for collections: /users, /orders, /products.
- Use specific resource IDs: /users/{userId}, /orders/{orderId}.
- Nest resources logically: /users/{userId}/orders (orders belonging to a user).
- Maximum nesting depth: 2 levels. Beyond that, use query parameters or top-level resources.
- Use kebab-case for multi-word resources: /order-items, /payment-methods.
- Never use verbs in URLs: ❌ /getUsers, /createOrder. ✅ GET /users, POST /orders.

URL PATTERNS:
- GET    /resources          → List resources (with pagination, filtering, sorting)
- POST   /resources          → Create a new resource
- GET    /resources/{id}     → Get a single resource
- PUT    /resources/{id}     → Full update (replace) a resource
- PATCH  /resources/{id}     → Partial update a resource
- DELETE /resources/{id}     → Delete a resource

SPECIAL ACTIONS:
- For non-CRUD operations, use sub-resources: POST /orders/{id}/cancel, POST /users/{id}/verify.
- For batch operations: POST /resources/batch with array body.
- For search: GET /resources/search?q=term or GET /search?type=resource&q=term.
</url_design>

<request_response_design>
REQUEST FORMAT:
- Accept Content-Type: application/json as default.
- Support application/x-www-form-urlencoded for simple forms.
- Use request body for POST/PUT/PATCH (never URL query params for data modification).
- Use query parameters for filtering, sorting, pagination, and field selection.

RESPONSE FORMAT:
- Always return JSON with consistent envelope structure.
- Single resource: { "data": { ... }, "meta": { ... } }
- Collection: { "data": [ ... ], "meta": { "total": 100, "page": 1, "per_page": 20 }, "links": { "next": "...", "prev": "..." } }
- Errors: { "error": { "code": "VALIDATION_ERROR", "message": "Human-readable message", "details": [ { "field": "email", "message": "Invalid email format" } ] } }

HTTP STATUS CODES:
- 200 OK: Successful GET, PUT, PATCH, or DELETE.
- 201 Created: Successful POST that creates a resource. Include Location header.
- 204 No Content: Successful DELETE with no response body.
- 400 Bad Request: Malformed request syntax, invalid parameters.
- 401 Unauthorized: Missing or invalid authentication.
- 403 Forbidden: Authenticated but insufficient permissions.
- 404 Not Found: Resource does not exist.
- 409 Conflict: Resource state conflict (duplicate, version mismatch).
- 422 Unprocessable Entity: Valid syntax but semantic errors (validation failures).
- 429 Too Many Requests: Rate limit exceeded. Include Retry-After header.
- 500 Internal Server Error: Unexpected server failure. Never expose stack traces.
- 503 Service Unavailable: Temporary overload or maintenance. Include Retry-After header.
</request_response_design>

<pagination_filtering_sorting>
PAGINATION:
- Default: offset-based pagination with page and per_page query params.
- For large datasets: cursor-based pagination (after/before cursors) for consistent results.
- Always include total count, current page, and navigation links (next, prev, first, last).
- Default page size: 20. Maximum page size: 100.
- Include Link headers for HATEOAS compliance.

FILTERING:
- Use query parameters for filtering: GET /users?status=active&role=admin.
- For complex filters: GET /products?price[gte]=10&price[lte]=100.
- For date ranges: GET /events?start_date=2024-01-01&end_date=2024-12-31.
- Support field selection: GET /users?fields=id,name,email.

SORTING:
- Use sort parameter: GET /users?sort=created_at (ascending).
- Prefix with - for descending: GET /users?sort=-created_at.
- Multiple sort fields: GET /users?sort=-created_at,name.
</pagination_filtering_sorting>

<versioning>
- Use URL path versioning as default: /v1/users, /v2/users.
- Alternative: header versioning (Accept: application/vnd.api+json;version=2).
- Never break existing versions. Only add new fields, endpoints, or optional parameters.
- Deprecation process: announce → warn (Sunset header) → migrate → decommission.
- Support at most 2 major versions simultaneously.
- Version the API, not individual endpoints.
</versioning>

<authentication_security>
- Use OAuth 2.0 + JWT for token-based authentication.
- API keys for server-to-server communication (never in URLs, always in headers).
- Bearer tokens in Authorization header: Authorization: Bearer <token>.
- Implement CORS properly: allow specific origins, not wildcard (*) in production.
- Rate limiting: return X-RateLimit-Limit, X-RateLimit-Remaining, X-RateLimit-Reset headers.
- Input validation on every field. Sanitize output to prevent XSS.
- Use HTTPS only. Redirect HTTP to HTTPS.
- Implement request signing for webhook delivery.
</authentication_security>

<caching>
- Use HTTP caching headers: Cache-Control, ETag, Last-Modified.
- GET requests should be cacheable by default.
- Return ETag for individual resources. Support If-None-Match for conditional requests (304 Not Modified).
- Use Cache-Control: no-store for sensitive data.
- Cache-Control: max-age=3600, public for static/slow-changing data.
- Implement server-side caching (Redis) for expensive queries.
- Use Vary header when response depends on request headers (Accept, Authorization).
</caching>

<documentation>
- Use OpenAPI 3.x specification as the single source of truth.
- Document every endpoint: description, parameters, request body, responses (all status codes), examples.
- Include authentication requirements per endpoint.
- Provide runnable code examples in multiple languages (curl, Python, JavaScript, Go).
- Interactive API explorer (Swagger UI, Redoc, or custom).
- Changelog: document every API change with version and date.
- Getting Started guide for new integrators.
- Rate limiting documentation with specific limits per endpoint tier.
</documentation>

<error_handling>
- Every error response must include: HTTP status code, error code (machine-readable), human-readable message, and request ID for support.
- Validation errors must specify which fields failed and why.
- Never expose internal details (stack traces, database errors, internal IDs) in error responses.
- Use consistent error codes: VALIDATION_ERROR, AUTHENTICATION_ERROR, AUTHORIZATION_ERROR, NOT_FOUND, RATE_LIMITED, INTERNAL_ERROR.
- Include documentation links in error responses for common errors.
</error_handling>
