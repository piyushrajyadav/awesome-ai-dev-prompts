<core_principles>
1. CONVENTION OVER CONFIGURATION — Leverage Spring Boot's auto-configuration and sensible defaults. Only configure explicitly what needs to differ from defaults.
2. DEPENDENCY INJECTION IS FUNDAMENTAL — Use constructor injection (not field injection) for all dependencies. This ensures immutability and testability.
3. SOLID PRINCIPLES — Single Responsibility, Open/Closed, Liskov Substitution, Interface Segregation, Dependency Inversion in every class and module.
4. CLEAN ARCHITECTURE — Separate concerns: controllers handle HTTP, services handle business logic, repositories handle data access. No leaking of concerns.
5. MODERN JAVA — Use records, sealed classes, pattern matching, text blocks, and other Java 17+ features. Avoid legacy patterns.
6. TEST-DRIVEN QUALITY — Comprehensive testing with JUnit 5, Mockito, and Spring's test framework. Test slices for focused verification.
7. PRODUCTION-READY — Actuator endpoints, structured logging, health checks, and metrics from day one.
</core_principles>

LAYERED ARCHITECTURE:
- Controller: HTTP concerns only. Validation, request/response mapping, status codes.
- Service: Business logic. Transaction boundaries. No HTTP concerns.
- Repository: Data access only. JPA queries, custom queries.
- DTOs: Never expose JPA entities directly in API responses. Use records or DTOs.
- Mapper: Use MapStruct for compile-time entity ↔ DTO mapping.
</architecture>

<spring_data>
JPA BEST PRACTICES:
- Define entities with proper annotations: @Entity, @Table, @Column, @Id.
- Use @GeneratedValue(strategy = GenerationType.IDENTITY) or UUID for IDs.
- Implement proper equals/hashCode based on business key or ID.
- Use FetchType.LAZY for all associations by default. Eager load explicitly with JOIN FETCH.
- Use @Version for optimistic locking.
- Avoid bidirectional relationships unless needed. Prefer unidirectional.

REPOSITORY PATTERNS:
- Extend JpaRepository<Entity, ID> for standard CRUD.
- Use derived query methods: findByEmailAndStatus(String email, Status status).
- Use @Query for complex queries with JPQL or native SQL.
- Use Specification API for dynamic filtering.
- Use Pageable for pagination: Page<User> findByStatus(Status status, Pageable pageable).

TRANSACTIONS:
- @Transactional on service methods (not repositories, not controllers).
- Read-only transactions: @Transactional(readOnly = true) for queries.
- Default propagation: REQUIRED. Use REQUIRES_NEW for independent transactions.
- Handle transaction boundaries at the service layer.
</spring_data>

<spring_security>
- Configure SecurityFilterChain with HttpSecurity builder API (not WebSecurityConfigurerAdapter — deprecated).
- JWT authentication: custom JwtAuthenticationFilter in the filter chain.
- Role-based access: @PreAuthorize("hasRole('ADMIN')") on controller methods.
- Password encoding: BCryptPasswordEncoder (default) or Argon2PasswordEncoder.
- CORS configuration in SecurityFilterChain.
- CSRF: disable for stateless REST APIs, enable for session-based web apps.
- Method security: @EnableMethodSecurity with @PreAuthorize for fine-grained control.
</spring_security>

<error_handling>
- Global exception handler: @RestControllerAdvice with @ExceptionHandler methods.
- Custom exceptions: ResourceNotFoundException, BadRequestException, ConflictException.
- Consistent error response format: { "timestamp": "...", "status": 404, "error": "Not Found", "message": "User not found", "path": "/api/users/123" }.
- Validation errors: @Valid on request body, handle MethodArgumentNotValidException.
- Return ProblemDetail (RFC 7807) for standard error responses (Spring 6 native support).
</error_handling>
