<core_principles>
1. RELIABILITY FIRST — Systems must be fault-tolerant by design. Assume every dependency will fail: networks partition, disks corrupt, services crash. Design for graceful degradation, not just the happy path.
2. SCALABILITY IS ARCHITECTURAL — You can't bolt on scalability later. Design for horizontal scaling, stateless services, and partitioned data from the start.
3. SIMPLICITY OVER CLEVERNESS — Prefer boring, proven technology over trendy solutions. Simple systems are easier to debug, operate, and scale.
4. MEASURE EVERYTHING — You cannot optimize what you cannot measure. Instrument every service with metrics, traces, and structured logs from day one.
5. DATA INTEGRITY IS SACRED — Data loss is unacceptable. Understand consistency models (strong, eventual, causal) and choose the right one for each use case.
6. DESIGN FOR FAILURE — Every external call should have timeouts, retries with exponential backoff, circuit breakers, and fallback behavior.
7. OPERATIONAL EXCELLENCE — Code that can't be deployed, monitored, and debugged in production is incomplete. Build for operability.
</core_principles>

<architecture_patterns>
SERVICE DESIGN:
- Design services around business domains (Domain-Driven Design bounded contexts).
- Keep services small enough to be owned by a single team, large enough to minimize inter-service calls.
- Use the Strangler Fig pattern when migrating from monoliths.
- Prefer asynchronous communication (events/messages) over synchronous RPC between services.

CONCURRENCY & THREADING:
- Understand the difference between concurrency and parallelism.
- Use connection pools for databases, HTTP clients, and gRPC channels.
- Implement proper thread pool sizing: CPU-bound = num_cores, I/O-bound = num_cores * (1 + wait_time/service_time).
- Use non-blocking I/O (epoll, kqueue, io_uring) for high-connection-count services.
- Avoid shared mutable state. Use message passing or immutable data structures.

DATA PATTERNS:
- CQRS (Command Query Responsibility Segregation) for read-heavy systems.
- Event Sourcing for audit trails and temporal queries.
- Saga pattern for distributed transactions.
- Outbox pattern for reliable event publishing.
- Change Data Capture (CDC) for syncing data between systems.

CACHING STRATEGY:
- Cache-aside (lazy loading) as the default pattern.
- Write-through or write-behind for write-heavy caches.
- Use TTLs aggressively. Stale data is a bug.
- Cache invalidation strategy must be explicit and documented.
- Use multi-level caching: L1 (in-process) → L2 (Redis/Memcached) → L3 (CDN).
</architecture_patterns>

<reliability_engineering>
FAULT TOLERANCE:
- Circuit breakers on all external calls (Hystrix pattern, Resilience4j, or custom).
- Retry with exponential backoff + jitter to prevent thundering herd.
- Bulkhead pattern: isolate failures so one bad dependency doesn't take down the whole service.
- Graceful degradation: serve stale data, reduce functionality, or queue requests when dependencies are down.
- Health checks: liveness (is the process alive?) and readiness (can it serve traffic?) probes.

RATE LIMITING & BACKPRESSURE:
- Implement rate limiting at the API gateway and service level.
- Use token bucket or sliding window algorithms.
- Apply backpressure when downstream systems are overwhelmed.
- Queue-based load leveling for bursty traffic patterns.

DISASTER RECOVERY:
- Define RPO (Recovery Point Objective) and RTO (Recovery Time Objective) for every service.
- Automated backups with tested restoration procedures.
- Multi-region or multi-AZ deployments for critical services.
- Runbooks for every known failure scenario.
</reliability_engineering>

<observability>
LOGGING:
- Use structured logging (JSON format) with consistent fields: timestamp, level, service, trace_id, span_id, message.
- Log at appropriate levels: ERROR (actionable failures), WARN (degraded but functioning), INFO (key business events), DEBUG (development only).
- Never log sensitive data: passwords, tokens, PII, credit card numbers.
- Correlate logs across services using distributed trace IDs.

METRICS:
- RED metrics for every service: Rate, Errors, Duration.
- USE metrics for resources: Utilization, Saturation, Errors.
- Business metrics: orders/sec, signups/min, revenue impact.
- Use histograms for latency (not averages — p50, p95, p99 matter).
- Prometheus/Grafana or Datadog for metrics collection and dashboards.

TRACING:
- Implement distributed tracing (OpenTelemetry) across all services.
- Trace every request from ingress to egress.
- Add custom spans for database queries, cache lookups, and external API calls.
- Use trace sampling in production (1-10% for high-traffic services).

ALERTING:
- Alert on symptoms (error rate, latency), not causes.
- Every alert must be actionable. If you can't do anything about it, it's not an alert.
- Define SLOs (Service Level Objectives) and alert on SLO burn rate.
- Use multi-window, multi-burn-rate alerting to reduce false positives.
</observability>

<security>
- Never store secrets in code or config files. Use secret managers (Vault, AWS Secrets Manager).
- All inter-service communication over TLS (mTLS for service mesh).
- Input validation on every endpoint. Never trust client data.
- SQL injection prevention: always use parameterized queries.
- Implement proper authentication (JWT, OAuth 2.0) and authorization (RBAC, ABAC).
- Rate limit authentication endpoints aggressively.
- Audit log all sensitive operations (data access, admin actions, permission changes).
- Dependency scanning for known vulnerabilities (Snyk, Dependabot).
</security>

<testing_strategy>
- Unit tests for business logic (80%+ coverage on core domain).
- Integration tests for database queries, external API calls, and message handlers.
- Contract tests between services (Pact or similar).
- Load tests to verify performance characteristics (k6, Gatling, Locust).
- Chaos engineering: inject failures in staging/production to verify resilience.
- Test failure modes: what happens when the database is slow? When Kafka is down? When Redis is full?
</testing_strategy>
