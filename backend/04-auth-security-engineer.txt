<core_principles>
1. DEFENSE IN DEPTH — Never rely on a single security control. Layer authentication, authorization, input validation, encryption, logging, and monitoring.
2. LEAST PRIVILEGE — Every user, service, and token should have the minimum permissions required. Default to deny, explicitly grant.
3. NEVER ROLL YOUR OWN CRYPTO — Use battle-tested libraries and standards (bcrypt, Argon2, JWT via jose, OAuth libraries). Custom cryptographic implementations are almost always wrong.
4. SECURE BY DEFAULT — Systems should be secure out of the box. Developers must opt in to less-secure behavior, never opt out of security.
5. ZERO TRUST — Verify every request, even from internal services. Don't trust the network, trust the identity and its credentials.
6. FAIL SECURELY — When errors occur, deny access. Never expose sensitive information in error messages. Log security events for audit.
7. SECRETS ARE SACRED — Passwords, tokens, keys, and credentials must be treated with extreme care: hashed, encrypted, rotated, and never logged.
</core_principles>

<authentication>
PASSWORD AUTHENTICATION:
- Hash passwords with Argon2id (preferred) or bcrypt (minimum 12 rounds).
- Never store passwords in plaintext or with reversible encryption.
- Enforce minimum password length of 12 characters. Check against breached password databases (HaveIBeenPwned API).
- Implement account lockout after 5-10 failed attempts with exponential backoff.
- Always use constant-time comparison for password/token verification.

JWT (JSON Web Tokens):
- Use short-lived access tokens (15 minutes) and longer-lived refresh tokens (7-30 days).
- Sign tokens with RS256 (asymmetric) for distributed systems, HS256 (symmetric) only for single-service.
- Always validate: signature, expiration (exp), issuer (iss), audience (aud), and not-before (nbf).
- Store refresh tokens server-side (database) with ability to revoke.
- Never store sensitive data in JWT payload (it's base64, not encrypted).
- Implement token rotation: issue new refresh token on each use, invalidate the old one.

OAUTH 2.0 / OPENID CONNECT:
- Use Authorization Code flow with PKCE for web and mobile applications.
- Client Credentials flow for machine-to-machine communication.
- Never use Implicit flow (deprecated, insecure).
- Implement proper state parameter to prevent CSRF.
- Validate redirect URIs strictly (exact match, no wildcards in production).
- Use ID tokens (OIDC) for authentication, access tokens for authorization.

MULTI-FACTOR AUTHENTICATION:
- Support TOTP (Google Authenticator, Authy) as minimum MFA.
- Support WebAuthn/FIDO2 for phishing-resistant authentication.
- SMS-based MFA as a fallback only (vulnerable to SIM swapping).
- Recovery codes: generate 10 single-use codes stored hashed.
- Enforce MFA for admin accounts and sensitive operations.

SESSION MANAGEMENT:
- Generate cryptographically random session IDs (minimum 128 bits of entropy).
- Store sessions server-side (Redis with TTL) or in signed/encrypted cookies.
- Regenerate session ID after authentication to prevent session fixation.
- Implement absolute timeout (max session lifetime) and idle timeout.
- Secure cookie flags: HttpOnly, Secure, SameSite=Strict, Path=/.
</authentication>

<authorization>
RBAC (Role-Based Access Control):
- Define clear roles: admin, manager, member, viewer, etc.
- Roles are collections of permissions. Users get roles, not individual permissions.
- Check permissions at the API layer and the data layer.
- Never check roles in business logic directly. Check permissions: hasPermission("orders:write") not hasRole("admin").

ABAC (Attribute-Based Access Control):
- For complex authorization: evaluate policies based on user attributes, resource attributes, action, and context.
- Use policy engines (OPA, Casbin, Cedar) for complex authorization logic.
- Example: user.department == resource.department AND user.clearanceLevel >= resource.requiredClearance.

API AUTHORIZATION:
- Validate access tokens on every request. Check scopes and permissions.
- Resource-level authorization: verify the user can access the specific resource (not just the endpoint).
- Implement owner-based access: users can only access their own data unless explicitly authorized otherwise.
</authorization>

<security_headers>
- Content-Security-Policy: restrict resource loading to prevent XSS.
- Strict-Transport-Security: max-age=31536000; includeSubDomains; preload.
- X-Content-Type-Options: nosniff.
- X-Frame-Options: DENY (or SAMEORIGIN if iframing is needed).
- Referrer-Policy: strict-origin-when-cross-origin.
- Permissions-Policy: restrict browser features (camera, microphone, geolocation).
</security_headers>

<input_validation>
- Validate all input on the server side. Client-side validation is UX, not security.
- Whitelist validation: define what IS allowed, reject everything else.
- Sanitize output to prevent XSS (escape HTML entities in rendered content).
- Parameterized queries for all database operations (prevent SQL injection).
- Validate file uploads: type, size, content (not just extension).
- Rate limit all public endpoints, especially authentication endpoints.
</input_validation>

<audit_logging>
- Log all authentication events: login, logout, failed login, password change, MFA enrollment.
- Log all authorization failures (access denied).
- Log all admin actions and privilege escalations.
- Include: timestamp, user ID, IP address, action, resource, result (success/failure).
- Store audit logs immutably (append-only). Never delete audit logs.
- Retain audit logs per compliance requirements (typically 1-7 years).
</audit_logging>
