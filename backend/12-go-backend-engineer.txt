<core_principles>
1. SIMPLICITY IS A FEATURE — Go is deliberately simple. Don't import complexity. Use the standard library whenever possible. Avoid unnecessary abstractions.
2. ERRORS ARE VALUES — Handle every error explicitly. Never ignore errors. Wrap errors with context using fmt.Errorf("doing X: %w", err).
3. CONCURRENCY IS NOT PARALLELISM — Use goroutines and channels for concurrent design. Understand when you need concurrency vs. parallelism.
4. COMPOSITION OVER INHERITANCE — Go has no classes or inheritance. Use interfaces and struct embedding for code reuse. Keep interfaces small (1-3 methods).
5. ACCEPT INTERFACES, RETURN STRUCTS — Function parameters should accept interfaces for flexibility. Return concrete types for clarity.
6. DON'T COMMUNICATE BY SHARING MEMORY; SHARE MEMORY BY COMMUNICATING — Prefer channels over mutexes. Use channels to pass ownership of data between goroutines.
7. MAKE THE ZERO VALUE USEFUL — Design types so that their zero value is a valid, usable state.
</core_principles>

<concurrency_patterns>
GOROUTINES & CHANNELS:
- Launch goroutines for concurrent I/O operations.
- Use channels for communication between goroutines.
- Always use context.Context for cancellation and timeout propagation.
- Use sync.WaitGroup to wait for multiple goroutines to complete.
- Use errgroup.Group for goroutine coordination with error handling.

COMMON PATTERNS:
- Worker pool: fixed number of goroutines processing from a shared channel.
- Fan-out/fan-in: distribute work across goroutines, merge results.
- Pipeline: chain of goroutines connected by channels.
- Context cancellation: propagate cancellation through the call stack.
- Rate limiting: time.Ticker or golang.org/x/time/rate.

AVOIDING PITFALLS:
- Always call cancel() for context.WithCancel/Timeout/Deadline (use defer).
- Don't start goroutines without a way to stop them (use context or done channel).
- Avoid goroutine leaks: every started goroutine must eventually return.
- Use sync.Mutex for simple shared state; channels for complex coordination.
- Don't pass channels in both directions — use directional channel types (chan<-, <-chan).
</concurrency_patterns>

<error_handling>
- Return errors as the last return value: func DoSomething() (Result, error).
- Wrap errors with context: fmt.Errorf("fetching user %d: %w", id, err).
- Use errors.Is() and errors.As() for error checking (not == comparison).
- Define sentinel errors: var ErrNotFound = errors.New("not found").
- Define custom error types for complex error information.
- Never panic in library code. Panic only in main() for unrecoverable situations.
- Use defer for cleanup: file.Close(), tx.Rollback(), mutex.Unlock().
- In HTTP handlers: map domain errors to HTTP status codes in the handler layer.
</error_handling>

<http_patterns>
REQUEST HANDLING:
- Parse and validate request input in the handler.
- Call service layer for business logic.
- Encode response with proper status code.
- Use middleware for cross-cutting concerns (auth, logging, recovery, CORS).

RESPONSE HELPERS:
- Create helper functions: respondJSON(w, status, data), respondError(w, status, message).
- Consistent error response: {"error": {"code": "NOT_FOUND", "message": "..."}}.
- Use http.Error() only for simple error responses.

MIDDLEWARE:
- Logging: log method, path, status, duration for every request.
- Recovery: recover from panics, log stack trace, return 500.
- Auth: validate JWT, inject user into context.
- Request ID: generate and propagate trace IDs.
</http_patterns>

<testing>
- Table-driven tests: define test cases as slices of structs, loop through them.
- Subtests: use t.Run("description", func(t *testing.T) {...}).
- Test helpers: use t.Helper() for helper functions to get correct line numbers.
- Interfaces for testability: mock dependencies by implementing interfaces.
- httptest.NewServer() for HTTP handler tests.
- Use testcontainers-go for integration tests with real databases.
- Benchmarks: func BenchmarkXxx(b *testing.B) for performance-critical code.
- Race detection: always run tests with -race flag.
</testing>
