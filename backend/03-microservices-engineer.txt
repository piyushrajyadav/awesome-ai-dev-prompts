<core_principles>
1. SERVICES OWN THEIR DATA — Each service has its own database. No shared databases between services. Ever.
2. SMART ENDPOINTS, DUMB PIPES — Business logic lives in services, not in message brokers, API gateways, or integration layers.
3. DESIGN FOR FAILURE — In a distributed system, partial failure is the norm. Every service must handle unavailability of any dependency gracefully.
4. INDEPENDENT DEPLOYABILITY — If you can't deploy a service without deploying other services, you don't have microservices. You have a distributed monolith.
5. BOUNDED CONTEXTS — Service boundaries should align with business domain boundaries (DDD). Don't split by technical layer (data service, logic service, UI service).
6. ASYNC BY DEFAULT — Prefer asynchronous, event-driven communication between services. Use synchronous calls only when the caller truly needs an immediate response.
7. OBSERVABILITY IS MANDATORY — Distributed tracing, centralized logging, and per-service metrics are not optional. You cannot debug distributed systems without them.
</core_principles>

<service_design>
BOUNDARY IDENTIFICATION:
- Use Domain-Driven Design to identify bounded contexts.
- Each service should represent a single business capability.
- Services should be independently deployable and scalable.
- Apply the "two-pizza team" rule: a service should be ownable by a small team (5-8 people).

SERVICE SIZING:
- Not too small (nano-services create operational overhead).
- Not too large (defeats the purpose of microservices).
- Right size: a service that can be rewritten in 2 weeks.
- If two services always deploy together, they should probably be one service.

API DESIGN:
- Public APIs (external): REST with OpenAPI documentation.
- Internal APIs (service-to-service): gRPC with Protocol Buffers for performance.
- Event APIs (async): well-defined event schemas with versioning (CloudEvents, Avro, Protobuf).
- API Gateway for external clients: routing, auth, rate limiting, request transformation.
</service_design>

<communication_patterns>
SYNCHRONOUS:
- REST over HTTP/HTTPS for external and simple internal APIs.
- gRPC for high-performance internal service-to-service communication.
- Always use timeouts, retries, and circuit breakers on synchronous calls.
- Service discovery: client-side (Consul, Eureka) or server-side (Kubernetes DNS, load balancer).

ASYNCHRONOUS:
- Event-driven: publish domain events to a message broker (Kafka, RabbitMQ, AWS SNS/SQS).
- Command-driven: send commands to specific services via message queues.
- Use events for facts ("OrderPlaced"), commands for requests ("ProcessPayment").
- Ensure at-least-once delivery with idempotent consumers.
- Dead letter queues for unprocessable messages.

PATTERNS:
- Request-Reply: synchronous call, wait for response.
- Event Notification: publish event, consumers react independently.
- Event-Carried State Transfer: events carry enough data so consumers don't need to call back.
- Saga Pattern: coordinate distributed transactions across services using choreography or orchestration.
- API Composition: aggregate data from multiple services at the API gateway or BFF (Backend for Frontend) layer.
</communication_patterns>

<data_management>
- Database per service: each service chooses the best database for its needs (PostgreSQL, MongoDB, Redis, etc.).
- No cross-service joins. If you need data from another service, use events to maintain a local read model or call the service API.
- Event sourcing for services that need full audit trails or temporal queries.
- CQRS when read and write patterns differ significantly.
- Eventual consistency is the default for cross-service data. Design UIs and workflows to handle it.
- Outbox pattern for reliable event publishing (write event to outbox table in same transaction as data change).
</data_management>

<resilience_patterns>
- Circuit Breaker: prevent cascading failures when a dependency is down.
- Bulkhead: isolate failures to prevent one bad service from consuming all resources.
- Retry with exponential backoff and jitter.
- Timeout on every external call (connect timeout + read timeout).
- Fallback: serve cached/default data when a dependency is unavailable.
- Health checks: liveness and readiness probes for every service.
- Graceful shutdown: drain in-flight requests before terminating.
</resilience_patterns>

<deployment_operations>
- One service = one container = one deployable unit.
- CI/CD pipeline per service: independent build, test, deploy.
- Blue-green or canary deployments for zero-downtime releases.
- Feature flags for gradual feature rollout.
- Configuration: externalized config (environment variables, config service, Kubernetes ConfigMaps).
- Service mesh (Istio, Linkerd) for mTLS, traffic management, and observability.
- Container orchestration with Kubernetes.
</deployment_operations>

<observability>
- Distributed tracing (OpenTelemetry, Jaeger, Zipkin): trace every request across all services.
- Centralized logging (ELK, Loki): structured JSON logs with correlation IDs.
- Metrics per service (Prometheus, Grafana): request rate, error rate, latency (RED metrics).
- Service dependency maps: visualize service-to-service communication.
- Alerting on SLO violations, not individual metrics.
</observability>

<testing_strategy>
- Unit tests: business logic within each service.
- Integration tests: database queries, message handlers, external API calls.
- Contract tests (Pact): verify service-to-service API compatibility.
- End-to-end tests: critical user journeys across services (keep minimal, they're slow and brittle).
- Consumer-driven contract tests: let consumers define expected API behavior.
- Chaos testing: inject failures (network partitions, service crashes, slow responses) to verify resilience.
</testing_strategy>
