<core_principles>
1. PYDANTIC IS YOUR BACKBONE — Every request body, response, query parameter, and config value is a Pydantic model. Type safety and validation are automatic.
2. DEPENDENCY INJECTION FOR EVERYTHING — Database sessions, auth, services, config — inject them all through FastAPI's Depends() system.
3. ASYNC BY DEFAULT — Use async def for all endpoints. Use async database drivers and HTTP clients. Never block the event loop.
4. SCHEMA-FIRST THINKING — Define your Pydantic models (schemas) first, then implement the logic. The schema IS the API contract.
5. AUTOMATIC DOCUMENTATION — Your OpenAPI docs should be comprehensive enough for any developer to integrate without additional documentation.
6. RESPONSE MODELS ARE SECURITY — Use response_model to explicitly control what data is returned. Never accidentally expose internal fields.
7. TEST WITH THE CLIENT — Use TestClient (httpx) for integration tests that validate the full request lifecycle.
</core_principles>

<pydantic_patterns>
REQUEST SCHEMAS:
- Use Field() for validation constraints: Field(min_length=1, max_length=100).
- Use model_validator for cross-field validation.
- Use Annotated types for reusable validations: Email = Annotated[str, Field(pattern=r"^[\w.-]+@[\w.-]+\.\w+$")].

RESPONSE SCHEMAS:
- Separate create, update, and response schemas. Never reuse input as output.
- Use model_config = ConfigDict(from_attributes=True) for ORM model conversion.
- Exclude sensitive fields (password_hash, internal_id) from response schemas.

CONFIG:
- Use BaseSettings with .env file support for all configuration.
- Validate config at startup: if required env vars are missing, fail fast.
- Use nested models for grouped settings (DatabaseSettings, RedisSettings, AuthSettings).

COMPUTED FIELDS:
- Use @computed_field for derived values in response models.
- Use @field_validator for custom validation logic.
- Use @field_serializer for custom serialization.
</pydantic_patterns>

<dependency_injection>
DATABASE DEPENDENCIES:
- async def get_db() -> AsyncGenerator[AsyncSession, None]: yield session, close on finish.
- Inject database session into repositories/services: Depends(get_db).

AUTH DEPENDENCIES:
- get_current_user: extract and validate JWT from Authorization header.
- require_role(role): factory that returns a dependency checking the user's role.
- Chain dependencies: get_current_active_user depends on get_current_user.

SERVICE DEPENDENCIES:
- Create factory functions: def get_user_service(db: AsyncSession = Depends(get_db)) -> UserService.
- Services receive their dependencies via constructor injection.
- Override dependencies in tests: app.dependency_overrides[get_db] = get_test_db.

SCOPED DEPENDENCIES:
- Use yield dependencies for setup/teardown (database sessions, locks).
- Dependencies are cached per request by default (same dependency with same params = same instance).
</dependency_injection>

<async_database>
SQLALCHEMY ASYNC:
- Use create_async_engine with asyncpg or aiosqlite driver.
- Use async_sessionmaker for session creation.
- Use select() and execute() instead of legacy query API.
- Eager loading: use selectinload() or joinedload() to prevent N+1 queries.
- Always access lazy-loaded relationships within the session scope.

REPOSITORY PATTERN:
- Each module has a repository handling all database operations.
- Repository methods are async and accept AsyncSession.
- Methods: create, get_by_id, get_many (with filters/pagination), update, delete.
- Return domain objects or ORM models, not raw rows.

MIGRATIONS:
- Alembic with async support for all schema migrations.
- Auto-generate migrations from model changes: alembic revision --autogenerate.
- Always review auto-generated migrations before applying.
- Include both upgrade and downgrade in every migration.
</async_database>

<error_handling>
CUSTOM EXCEPTIONS:
- Define hierarchy: AppException → NotFoundException, ConflictException, ForbiddenException.
- Each exception has: status_code, error_code (machine-readable), message (human-readable).
- Register exception handlers in main.py: @app.exception_handler(AppException).

ERROR RESPONSES:
- Consistent format: {"error": {"code": "NOT_FOUND", "message": "User not found", "details": []}}.
- Use FastAPI's HTTPException for simple cases.
- Use custom exception handlers for automatic formatting.
- Validation errors (422) include field-level details automatically from Pydantic.
</error_handling>

<performance>
- Use uvicorn with multiple workers (or gunicorn with uvicorn workers) for production.
- Connection pooling: configure pool_size and max_overflow on async engine.
- Use BackgroundTasks for non-critical operations (sending emails, logging analytics).
- Implement response caching with Redis for expensive queries.
- Use Streaming responses (StreamingResponse) for large data exports.
- Profile with py-spy or cProfile to identify bottlenecks.
</performance>

<testing>
- Use pytest with pytest-asyncio for async test support.
- Create a test client fixture with httpx.AsyncClient or TestClient.
- Use a test database (separate PostgreSQL database or SQLite in-memory).
- Override dependencies in tests for mocking.
- Test happy paths, validation errors, auth failures, and edge cases.
- Use factory fixtures for creating test data.
</testing>
