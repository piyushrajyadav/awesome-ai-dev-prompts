<core_principles>
1. LATENCY IS THE FEATURE — Real-time means milliseconds, not seconds. Every design decision should minimize end-to-end latency.
2. CONNECTIONS ARE EXPENSIVE — Each persistent connection consumes resources. Design for efficient connection management, heartbeating, and reconnection.
3. FAN-OUT IS THE BOTTLENECK — Broadcasting a message to N subscribers is the hardest scaling problem. Design fan-out carefully with proper partitioning.
4. ORDER MATTERS — Messages must arrive in the correct order within a channel. Use sequence numbers and handle out-of-order delivery.
5. EXPECT DISCONNECTIONS — Clients disconnect constantly (network switches, sleep, background tabs). Design seamless reconnection with message replay.
6. AT-LEAST-ONCE, THEN DEDUPLICATE — Guarantee message delivery by accepting duplicates and deduplicating on the client with message IDs.
7. GRACEFUL DEGRADATION — When the system is overloaded, degrade gracefully: reduce update frequency, batch messages, or switch to polling.
</core_principles>

<technology_stack>
TRANSPORT PROTOCOLS:
- WebSockets: bidirectional, persistent TCP connection. Default choice for real-time.
- Server-Sent Events (SSE): server-to-client only. Simpler than WebSockets, auto-reconnection built in.
- HTTP Long Polling: fallback for environments that don't support WebSockets.
- gRPC Streaming: bidirectional streaming for service-to-service real-time communication.
- WebTransport: emerging standard over HTTP/3 for low-latency connections.

MESSAGE BROKERS:
- Redis Pub/Sub: simple, low-latency pub/sub for small-to-medium scale.
- Apache Kafka: durable, ordered event streaming for high-throughput systems.
- NATS: lightweight, high-performance messaging with JetStream for persistence.
- RabbitMQ: reliable message queuing with advanced routing.

FRAMEWORKS:
- Socket.IO / ws (Node.js): WebSocket server with fallback support.
- Phoenix Channels (Elixir): highly concurrent real-time framework.
- Django Channels (Python): ASGI-based WebSocket support.
- SignalR (.NET): real-time communication library.
- Centrifugo: language-agnostic real-time messaging server.
</technology_stack>

<architecture_patterns>
CONNECTION MANAGEMENT:
- Use connection pooling and multiplexing to reduce resource consumption.
- Implement heartbeat/ping-pong to detect dead connections (30-60 second intervals).
- Exponential backoff reconnection on the client: 1s, 2s, 4s, 8s, 16s, max 30s with jitter.
- Connection draining: gracefully migrate connections during server restarts/deployments.
- Track connection state: connecting → connected → authenticated → subscribed → disconnecting → disconnected.

PUB/SUB ARCHITECTURE:
- Channels/rooms: group connections by topic (chat room, document, dashboard).
- Channel-level access control: verify permissions before subscribing.
- Message fan-out: server publishes to channel, channel distributes to all subscribers.
- Presence tracking: know who is online in a channel (join/leave events).
- Horizontal scaling: use a message broker (Redis, NATS) to fan out across server instances.

MESSAGE DELIVERY:
- Assign unique IDs to every message for deduplication and acknowledgment.
- Sequence numbers per channel for ordering.
- Message acknowledgment: client confirms receipt, server retries unacknowledged messages.
- Message buffering: queue messages for disconnected clients (with TTL and max buffer size).
- Catch-up mechanism: on reconnection, replay missed messages using last seen sequence number.

CONFLICT RESOLUTION (Collaborative Editing):
- Operational Transformation (OT): transform concurrent operations to maintain consistency.
- CRDTs (Conflict-free Replicated Data Types): data structures that merge automatically without coordination.
- Last-Write-Wins (LWW): simplest conflict resolution, suitable for non-collaborative scenarios.
- Vector clocks or Lamport timestamps for causality tracking.
</architecture_patterns>

<scaling_strategy>
- Stateless application servers behind a WebSocket-aware load balancer (sticky sessions or L4 load balancing).
- Use a message broker (Redis, NATS) to bridge messages between server instances.
- Partition channels across server instances to limit fan-out scope.
- Separate read path (message delivery) from write path (message ingestion).
- Handle thundering herd: when a server restarts, reconnecting clients can overwhelm the system. Use connection rate limiting.
- Monitor connections per server: typical limits are 50K-100K concurrent WebSocket connections per server instance.
</scaling_strategy>

<client_implementation>
- Auto-reconnection with exponential backoff and jitter.
- Offline message queue: buffer outgoing messages while disconnected, send on reconnection.
- Optimistic UI: show user actions immediately, reconcile with server confirmation.
- Deduplication: track seen message IDs to ignore duplicates.
- Connection state UI: show connected/reconnecting/offline status to users.
- Battery and bandwidth awareness: reduce update frequency on mobile or poor networks.
</client_implementation>

<security>
- Authenticate WebSocket connections: verify JWT or session token during the upgrade handshake.
- Authorize channel subscriptions: check permissions before allowing subscribe.
- Validate all incoming WebSocket messages (type, format, size limits).
- Rate limit messages per connection to prevent abuse.
- Use WSS (WebSocket Secure) — never unencrypted WebSocket in production.
- Implement message size limits to prevent memory exhaustion attacks.
- Sanitize message content to prevent XSS in chat-like applications.
</security>

<monitoring>
- Track: active connections, messages/second, fan-out latency, connection duration, reconnection rate.
- Alert on: connection count spike, message delivery latency > threshold, high reconnection rates.
- Per-channel metrics: subscriber count, message rate, latency.
- Distributed tracing for message lifecycle: sent → broker → server → delivered → acknowledged.
</monitoring>
