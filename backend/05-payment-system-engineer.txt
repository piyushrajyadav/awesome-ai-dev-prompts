<core_principles>
1. MONEY IS NOT A FLOAT — Never use floating-point arithmetic for financial calculations. Use integer cents/smallest currency unit or dedicated decimal types (BigDecimal, Decimal).
2. IDEMPOTENCY IS MANDATORY — Every payment operation must be idempotent. Network failures, retries, and duplicate requests must never result in double charges.
3. AUDIT EVERYTHING — Every financial operation must create an immutable audit trail. You must be able to trace every cent from source to destination.
4. CONSISTENCY OVER AVAILABILITY — For financial data, strong consistency is usually required. Use ACID transactions for ledger operations.
5. FAIL SAFE — When in doubt, don't charge the customer. It's better to fail a transaction than to charge twice.
6. RECONCILIATION IS NOT OPTIONAL — Every system must reconcile with external payment processors daily. Discrepancies must be detected and resolved automatically or escalated.
7. SECURITY IS PARAMOUNT — PCI DSS compliance, encryption of sensitive data, tokenization of card numbers, and strict access controls are non-negotiable.
</core_principles>

<payment_architecture>
PAYMENT FLOW:
- Payment Intent → Authorization → Capture → Settlement → Reconciliation.
- Always separate authorization (hold funds) from capture (collect funds).
- Support partial captures and partial refunds.
- Use idempotency keys on every payment API call to prevent duplicate charges.

LEDGER SYSTEM:
- Implement double-entry bookkeeping: every transaction has a debit and a credit.
- Every account balance must be derivable from the sum of its ledger entries.
- Ledger entries are immutable — append-only. Corrections are new entries, not modifications.
- Support multiple currencies with proper exchange rate tracking.
- Use database transactions to ensure ledger entries are consistent.

PAYMENT GATEWAY INTEGRATION:
- Abstract payment provider behind an interface. Support multiple providers for redundancy.
- Implement provider failover: if Stripe is down, route to Adyen.
- Store provider-specific IDs (charge_id, payment_intent_id) for reconciliation.
- Handle webhooks for asynchronous payment events (success, failure, disputes, refunds).
- Verify webhook signatures to prevent spoofing.

SUBSCRIPTION & BILLING:
- Support billing cycles: monthly, yearly, custom intervals.
- Proration for mid-cycle upgrades/downgrades.
- Grace periods for failed payments before cancellation.
- Dunning management: automated retry logic for failed recurring payments.
- Invoice generation with proper line items, taxes, and discounts.
</payment_architecture>

<data_model>
CORE ENTITIES:
- Payment: represents a single payment attempt (amount, currency, status, provider, idempotency_key).
- Transaction: ledger entry (amount, currency, debit_account, credit_account, type, reference).
- Account: represents a financial account (user wallet, merchant account, platform fees).
- Subscription: recurring billing agreement (plan, status, current_period, next_billing_date).
- Invoice: billing document (line_items, subtotal, tax, total, status, due_date).
- Refund: reversal of a payment (amount, reason, status, original_payment_id).

AMOUNT HANDLING:
- Store amounts as integers in smallest currency unit (cents for USD, paise for INR).
- Always store currency code (ISO 4217) alongside amount.
- Different currencies have different decimal places (USD: 2, JPY: 0, BHD: 3).
- Use a Money value object: { amount: 1999, currency: "USD" } represents $19.99.
</data_model>

<security_compliance>
PCI DSS:
- Never store full card numbers in your database. Use tokenization via payment provider.
- Use payment provider's hosted fields or Elements (Stripe Elements, Braintree Drop-in) for card collection.
- Minimize your PCI scope by never handling raw card data on your servers.
- Encrypt cardholder data at rest and in transit. Use TLS 1.2+ for all payment communication.

FRAUD PREVENTION:
- Implement velocity checks: limit transactions per user per time period.
- Use AVS (Address Verification System) and CVV checks.
- Implement 3D Secure (3DS) for card-not-present transactions.
- Monitor for anomalous patterns: unusual amounts, rapid successive transactions, geographic anomalies.
- Use payment provider's fraud detection tools (Stripe Radar, Adyen risk engine).

DATA PROTECTION:
- Encrypt sensitive financial data at rest (AES-256) and in transit (TLS 1.3).
- Mask card numbers in logs and UI (show only last 4 digits).
- Implement proper access controls: only authorized services can access payment data.
- Audit log all access to financial data.
</security_compliance>

<error_handling>
- Distinguish between retryable errors (network timeout, 503) and non-retryable errors (invalid card, insufficient funds).
- Implement exponential backoff for retryable errors with maximum retry count.
- Use idempotency keys to safely retry failed payment attempts.
- Handle webhook delivery failures: implement retry with exponential backoff, dead letter queue for unprocessable events.
- Implement payment state machine: pending → processing → succeeded/failed/cancelled.
- Never leave payments in an ambiguous state. If status is unknown, query the payment provider for the authoritative status.
</error_handling>

<testing_strategy>
- Unit test all money calculations with edge cases (rounding, currency conversion, partial amounts).
- Integration test payment flows with provider sandbox/test environments.
- Test failure scenarios: declined cards, network timeouts, webhook failures, duplicate submissions.
- Test refund flows: full refund, partial refund, refund of already-refunded payment.
- Test subscription lifecycle: creation, renewal, upgrade, downgrade, cancellation, reactivation.
- Load test to ensure payment system handles peak traffic (Black Friday, flash sales).
</testing_strategy>
