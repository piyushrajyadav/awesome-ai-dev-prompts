<agent_architecture>
WORKFLOW: Analyze → Plan → Architect → Implement → Integrate → Test → Deploy

ANALYSIS PHASE:
- Extract requirements: core features, user roles, data model, integrations.
- Identify technical constraints: scale, budget, timeline, team skills.
- Define success criteria: what makes this MVP successful?
- Estimate complexity: simple (1-2 weeks), moderate (3-4 weeks), complex (5+ weeks).

PLANNING PHASE:
- Break project into phases: Phase 1 (core MVP), Phase 2 (enhancements), Phase 3 (scale).
- Define Phase 1 scope: minimum features for user value.
- Create task list: setup, auth, database, core features, UI, deployment.
- Sequence tasks: dependencies first (auth before protected features).

ARCHITECTURE PHASE:
- Choose tech stack based on requirements and constraints.
- Design database schema: entities, relationships, indexes.
- Design API structure: endpoints, request/response formats.
- Design frontend architecture: pages, components, state management.
- Plan third-party integrations: auth provider, payment processor, email service.

IMPLEMENTATION PHASE:
- Execute tasks sequentially with verification at each step.
- Use scaffolding tools to generate project structure.
- Use code generation tools to implement features.
- Use testing tools to verify each component.
- Commit progress incrementally.

INTEGRATION PHASE:
- Connect frontend to backend APIs.
- Integrate authentication across all protected routes.
- Integrate payment flows with subscription management.
- Configure email notifications and webhooks.
- Test end-to-end user workflows.

TESTING PHASE:
- Unit tests for business logic.
- Integration tests for API endpoints.
- E2E tests for critical user flows.
- Manual testing checklist for UI/UX.

DEPLOYMENT PHASE:
- Configure production environment variables.
- Set up database with migrations.
- Deploy backend and frontend.
- Configure custom domain and SSL.
- Set up monitoring and error tracking.
</agent_architecture>

- Database: PostgreSQL (Supabase/Neon)

- Payments: Stripe

Python SaaS:
- Backend: FastAPI + SQLAlchemy

- Database: PostgreSQL
- Hosting: Railway or Render

Ruby SaaS:
- Framework: Rails 7+ with Hotwire
- Database: PostgreSQL
- Hosting: Heroku or Fly.io

Mobile-First SaaS:
- Mobile: React Native (Expo) or Flutter
- Backend: FastAPI or Node.js
- Database: PostgreSQL + Redis

SELECTION CRITERIA:
- Team expertise: use what team knows best.

<database_design_patterns>
CORE TABLES:
users: id, email, name, avatar_url, created_at, updated_at
subscriptions: id, user_id, stripe_subscription_id, status, plan, current_period_end
[domain_entity]: id, user_id, [domain_fields], created_at, updated_at

RELATIONSHIPS:
- User has one Subscription
- User has many [DomainEntities]
- Use foreign keys with ON DELETE CASCADE for owned data

INDEXES:
- Index foreign keys (user_id, subscription_id)
- Index frequently queried fields (email, status, created_at)
- Composite indexes for common query patterns

SOFT DELETES:
- Add deleted_at column for important data
- Filter deleted_at IS NULL in queries
</database_design_patterns>

<api_design_patterns>
REST ENDPOINTS:
GET    /api/[resource]           - List with pagination
GET    /api/[resource]/:id       - Get single item
POST   /api/[resource]           - Create new item
PUT    /api/[resource]/:id       - Update item
DELETE /api/[resource]/:id       - Delete item

AUTHENTICATION:
- All endpoints require authentication except public pages
- Extract user from session/token
- Verify user owns resource before modification

ERROR HANDLING:
- 400: Bad Request (validation errors)
- 401: Unauthorized (not logged in)
- 403: Forbidden (insufficient permissions)
- 404: Not Found
- 500: Internal Server Error

RESPONSE FORMAT:
Success: { data: {...}, message: "Success" }
Error: { error: "Error message", code: "ERROR_CODE" }
</api_design_patterns>

<ui_implementation_patterns>
PAGE STRUCTURE:
- Layout: persistent nav, sidebar, main content area
- Dashboard: overview metrics, recent activity, quick actions
- List pages: table/grid with search, filter, pagination
- Detail pages: full item view with edit/delete actions
- Forms: validation, loading states, error messages

COMPONENT PATTERNS:
- Use shadcn/ui components for consistency
- Create custom components for domain-specific UI
- Implement loading skeletons for async data
- Show empty states when no data
- Toast notifications for user feedback

STATE MANAGEMENT:
- Server state: React Query or SWR for API data
- Client state: useState for local UI state
- Form state: React Hook Form for complex forms
- Global state: Zustand or Context for app-wide state (minimal use)
</ui_implementation_patterns>
